# 常见问题排查

> 适用范围：Basalt OS 清算分账系统
> 来源文档：Basalt OS 核心业务逻辑与实现方案 v6.0、网商银行集中收银分账接口整理 v3.1

---

## 概述

本文档汇总 Basalt OS 清算分账系统中的常见问题及排查方法，覆盖商户入驻、清算批次、协议签约、付款提现、退款、退保、通知消息等全部业务场景。排查思路遵循"先看接口返回码 → 再看业务状态字段 → 最后核查前置条件"的三步法。

### 通用结果码速查

所有网商接口返回的 `RespInfo` 中包含 `ResultStatus` 字段，通用含义如下：

| ResultStatus | 含义 | 说明 |
|-------------|------|------|
| **S** | 成功 | 接口调用成功（但不代表业务成功，需进一步看业务状态字段） |
| **F** | 失败 | 接口调用失败，需查看 `ResultCode` 和 `ResultMsg` 获取原因 |
| **U** | 处理中 | 异步处理中，需通过查询接口或等待通知接口获取最终结果 |

> **关键提醒**：`ResultStatus=S` 仅代表接口调用成功，不代表业务结果成功。例如入驻查询接口返回 S，但 `RegisterStatus` 可能是 0（审核中）、1（成功）、2（失败），需看具体业务字段。

## 操作步骤

### 1. 入驻/激活失败排查

#### 1.1 预入驻失败

**症状**：调用预入驻接口后，查询结果 `RegisterStatus=2`（失败）

**排查步骤**：

1. **检查返回的 FailReason 字段**：预入驻结果查询接口（`ant.mybank.merchantprod.merch.register.query`）会返回失败原因
2. **核查入参完整性**：
   - `OutTradeNo` 是否唯一（同一交易号被视为同一笔交易）
   - `OutMerchantId` 是否已被占用
   - `MerchantType` 是否正确：01-自然人、02-个体工商户、03-企业商户
   - 身份证图片（CertPhotoA/B）是否已通过图片上传接口获取URL
   - 企业商户是否提供了营业执照（LicensePhoto）和联行号（ContactLine）
3. **核查预入驻前置条件**：
   - 预入驻阶段绑卡只支持企业商户
   - 结算到银行卡时 BankCardNo 必填
   - 结算到支付宝时 LogonId 必填

#### 1.2 激活失败

**症状**：商户完成激活后，查询结果 `RegisterStatus=2`（失败）或一直无结果

**排查步骤**：

1. **确认商户是否已打开激活链接**：
   - 激活结果查询接口（`ant.mybank.merchantprod.merch.applet.register.query`）返回的 `MerchantId` 和 `RegisterStatus` 在用户未开始激活时**不存在**
   - 如果这两个键值不存在，说明商户还未开始激活流程
2. **检查激活链接是否正确**：
   - 线上环境：`alipays://platformapi/startapp?appId=2021001187624111&nbupdate=syncforce&query=outMchId%3D{outMchId}%26isv%3D{ISV_ORG_ID}%26app%3Dbkclfs%26t%3DACTIVATE`
   - 测试环境需额外添加 `&uid%3D{uid}` 参数
   - `{outMchId}` 和 `{ISV_ORG_ID}` 是否替换正确
3. **确认预入驻是否已审核通过**：激活必须在预入驻审核通过后才能进行
4. **查看通知接口是否收到结果**：检查是否收到预入驻激活结果通知（3.4接口）
5. **异步通知是否被正确处理**：平台需返回 `ResultStatus=S`，否则网商会持续重试

#### 1.3 小程序入驻失败

**症状**：商户通过小程序注册后，查询结果 `RegisterStatus=2`（失败）

**排查步骤**：

1. **检查注册通知**：查看是否收到商户注册通知接口（`ant.mybank.merchantprod.merch.notify`）回调，包含 `FailReason`
2. **确认注册链接格式**：注册链接中 `t=REGISTER`，激活链接中 `t=ACTIVATE`，不要混淆
3. **确认 outMchId 唯一性**：同一 outMchId 可重复生成链接，但入驻结果唯一

### 2. 清算批次提交失败排查

#### 2.1 三种清算结果含义

| 状态 | 含义 | 处理建议 |
|------|------|---------|
| **SUCCESS** | 全部订单清算成功 | 正常处理后续流程 |
| **PART_SUCCESS** | 部分成功 | 需下载 `VerifyFailDataFileUrl` 校验文件（默认三天有效期），确认每笔订单的具体结果 |
| **FAIL** | 全部失败 | 检查 `ErrorDesc` 错误信息，排查原因后可重新创建批次 |

#### 2.2 创建批次接口调用失败

**排查步骤**：

1. **检查前置条件**：
   - 补单账单文件是否已成功上传至SFTP（必须先上传文件再创建批次）
   - 是否上传了完结标志文件 `finish_${batchNo}.txt`
2. **检查入参**：
   - `BatchNo` 是否在平台维度下唯一（仅支持数字、字母、大小写字符）
   - `BatchType` 是否正确：`STMT_AVAILABLE_SUM_SHARE`（结算至余额并分账-汇总分账）
   - `FileUploadType` 是否为 `SFTP`
   - `AcquireId` 默认传 `DEFAULT`
3. **检查幂等**：网商基于 `IsvOrgId+BatchNo+AcquireId` 做幂等处理，重复提交会返回缓存结果

#### 2.3 上报失败（批次状态"上报失败"）

**排查步骤**：

1. 检查调用创建批次接口时的 `RespInfo` 返回信息
2. 确认SFTP文件路径是否正确：`upload/${isvOrgId}/${acquireId}/${batchNo}/merchant_trade_detail_${batchNo}.txt`
3. 确认清算文件格式是否正确：channel_type, trade_type, batch_no, mybank_order_no, merchant_id, trade_amount 等字段
4. 上报失败后可以选择重试或作废批次

#### 2.4 批次一直"清算中"不返回结果

**排查步骤**：

1. **检查通知接口**：确认批次完结通知接口（`ant.mybank.bkcloudfunds.billpay.batch.finish.notify`）是否正常接收
   - 通知频率：1,3,5,10,15,30,60,300,600,900,1800,3600（秒）
   - 平台需在10秒内返回成功响应
2. **主动查询**：调用批次查询接口（`ant.mybank.bkcloudbatch.batch.query`）手动查询状态
3. **检查通知回调地址**：确认平台通知接收服务是否正常运行
4. **注意**：系统设计原则是"通知+查询"双保险，通知可能丢失，定时查询可兜底

#### 2.5 PART_SUCCESS 处理

**排查步骤**：

1. 从批次完结通知中获取 `VerifyFailDataFileUrl`（校验失败信息下载地址）
2. 下载校验文件，逐笔确认每个订单的清算结果
3. 清算失败的订单可重新上送并纳入新的批次
4. **核心规则**：清算失败订单重提时必须沿用原始商户映射，不得动态更新商户映射，避免对账失败

### 3. 协议签约失败排查

#### 3.1 协议代扣签约失败

**症状**：商户在小程序签约后，平台未收到签约通知或查询合约状态异常

**排查步骤**：

1. **查询合约状态**：调用协议代扣授权状态查询接口（`ant.mybank.merchantprod.merchant.arrangement.info.query`）

   | 状态码 | 含义 | 处理方式 |
   |--------|------|---------|
   | NOT_EXIST | 不存在（未签约） | 引导商户重新签约 |
   | UN_VALID | 已申请未生效 | 检查签约流程是否完整 |
   | VALID | 已生效 | 正常状态，可发起代扣 |
   | INVALID_TO_BE_CONFIRM | 解约中待平台确认 | 需平台审核解约申请 |

2. **检查通知接口**：确认授权/解约通知接口（`ant.mybank.merchantprod.merchant.arrangement.info.notify`）是否正常接收
3. **联动查询**：收到通知后应立即调用查询接口再确认一次合约最新状态

#### 3.2 授权代付签约失败

**症状**：平台发起签约申请后，商户未确认或签约最终失败

**排查步骤**：

1. **检查签约申请返回**：
   - 同步接口（`ant.mybank.bkcloudfunds.protocol.sign.apply`）：检查 `Status` 字段（SUCCESS/FAIL）
   - 失败时查看 `ErrorCode` 和 `ErrorDesc`
   - 成功时确认是否拿到 `AuthSignUrl`（确认链接）
2. **确认链接是否正确发放给商户**：商户需通过 AuthSignUrl 打开小程序完成授权确认
3. **检查异步通知**：查看协议签约异步申请结果通知接口（`ant.mybank.bkcloudfunds.protocol.sign.apply.notify`）
   - 异步接口（>50收方）立即返回 `PROCESSING`，需等待通知或查询
4. **查询申请单状态**：调用查询申请单状态接口（`ant.mybank.bkcloudfunds.protocol.sign.apply.query`）
5. **超时自动确认**：
   - **仅工资代发（SALARY）场景**支持超时自动确认：商户48小时内未确认时网商自动确认通过
   - 其他场景不支持超时自动确认，必须商户手动确认
6. **如需取消**：可调用取消申请接口（`ant.mybank.bkcloudfunds.protocol.sign.apply.cancel`）取消待确认的申请单

#### 3.3 收款方银行卡限制

- 仅支持**银联储蓄卡**（仅中国大陆地区的储蓄卡）
- 不支持信用卡、境外银行卡
- 支付宝账号收款无此限制

### 4. 付款/提现状态一直 DEALING 的处理

#### 4.1 协议代扣付款状态 DEALING

**排查步骤**：

1. **主动查询**：调用授权资金划扣结果查询接口（`ant.mybank.bkcloudfunds.protocol.withhold.query`），传入 `WithholdNo`（划扣单号）
2. **检查通知**：确认授权资金划扣结果通知接口（`ant.mybank.bkcloudfunds.protocol.withhold.result.notify`）是否正常
3. **查看失败原因**：如查询返回失败状态，查看 `FailReason` 字段
4. **常见失败原因**：
   - 商户余额不足
   - 合约已失效/已解约
   - 商户被限权

#### 4.2 授权代付付款状态 DEALING

**排查步骤**：

1. **主动查询**：调用授权代付结果查询接口（`ant.mybank.bkcloudfunds.trade.operate.result.query`）
2. **检查通知**：确认授权代付结果通知接口（`ant.mybank.bkcloudfunds.trade.operate.result.notify`）是否正常
3. **联动查询**：收到通知后再用查询接口查一次付款单数据
4. **手动更新**：用户点击"更新付款状态"按钮，调用查询接口查询全部"处理中"状态的付款单

#### 4.3 提现状态一直 DEALING

**排查步骤**：

1. **主动查询**：调用单笔提现查询接口（`ant.mybank.bkcloudfunds.withdraw.query`）
   - Status 为 `DEALING` 表示处理中，需继续查询
   - 查询到 `SUCCESS` 或 `FAIL` 后为终态
2. **检查通知**：确认提现结果通知接口（`ant.mybank.bkcloudfunds.withdraw.notify`）是否正常
3. **确认是否完成短信验证**：
   - 提现申请后需商户完成短信验证
   - 提现确认接口（`ant.mybank.bkcloudfunds.withdraw.applyconfirm`）调用成功后才真正扣款
   - 提现申请阶段不会扣减余额
4. **检查 ErrorDesc**：提现查询接口返回的 `ErrorDesc` 字段会描述错误原因

#### 4.4 批量付款批次状态说明

| 状态 | 含义 |
|------|------|
| 处理中 | 系统解析中或网商处理中，仍有未返回结果的付款单 |
| 成功 | 批次内所有付款单全部成功 |
| 失败 | 全部解析失败或全部付款单失败 |
| 部分成功 | 部分成功部分失败 |

### 5. 退款失败的处理

**说明**：网商不需要退款单上送，系统中直接将退款订单状态变更为"已上送"即可，不调接口。退款在清算分账系统中主要通过**扣保和退保机制**处理。

#### 5.1 退款扣保相关排查

**扣保通知类型（只需处理3种）**：

| 类型 | 说明 | 处理优先级 | 备注 |
|------|------|-----------|------|
| 类型 1 | 余额不足扣保 | P0（核心场景） | 必须处理 |
| 类型 3 | 尾差扣保 | P2（极少触发） | 兜底场景 |
| 类型 5 | 退保结果通知 | P0（核心场景） | 必须处理 |
| ~~类型 2~~ | ~~明细单扣保~~ | - | **已失效** |
| ~~类型 4~~ | ~~尾差退还~~ | - | **无法使用，需人工处理** |

#### 5.2 扣保通知未收到

**排查步骤**：

1. **确认扣保是否发生**：批次完结通知中 `IsDepositPay=Y` 表示发生了结算单扣保，`DepositPayCnt` 为扣保商户数量
2. **检查通知接口**：确认补单保证金扣除&退回通知接口（`ant.mybank.bkcloudbatch.stmt.deposit.notify`）是否正常
3. **注意**：只有批次成功时才会进行结算单扣保统计，批次失败不会

#### 5.3 协议代扣退款

协议代扣退款接口（`ant.mybank.bkcloudfunds.protocol.withhold.refund.apply` / `.query`）不在60个核心接口范围内，如需接入请另行查阅协议代扣完整文档。

### 6. 通知消息重复的处理（幂等）

网商的异步通知接口均会重试，通知频率为 `1,3,5,10,15,30,60,300,600,900,1800,3600`（单位：秒），平台必须做好幂等处理。

#### 6.1 各通知接口的幂等键

| 通知接口 | 幂等键 | 说明 |
|---------|-------|------|
| 渠道结算入账通知 | `OrderNo` | 来账订单号去重 |
| 批次完结通知 | `BatchNo` + `BatchInstanceId` | 批次维度去重 |
| 扣保通知 | `DepositPayId` | 扣保单编号去重 |
| 退保通知 | `RefundDepositOutTradeNo` | 退保外部请求单号去重 |
| 协议代扣授权/解约通知 | `ArrangementNo` | 合约号去重 |
| 授权代付合同变更通知 | `AgreementNo` | 合约号去重 |
| 代扣结果通知 | `WithholdNo` | 划扣单号去重 |
| 代付结果通知 | 付款单号 | 付款单维度去重 |
| 提现结果通知 | `OutTradeNo` / `OrderNo` | 提现订单维度去重 |
| 商户注册通知 | `OutMerchantId` | 外部商户号去重 |
| 商户限权通知 | `ControlOrderNo` + `PunishDate` | 管控单号+管控时间排序 |

#### 6.2 幂等处理通用方案

1. **持久化通知消息**：将每条通知消息存储到数据库，以幂等键作为唯一索引
2. **返回成功响应**：平台需返回 `ResultStatus=S` 告知网商接收成功，否则网商会持续重试
3. **业务去重**：处理通知前检查该幂等键是否已处理，已处理则直接返回成功
4. **乱序处理**：对于商户限权通知，使用 `PunishDate`（管控时间）判断通知先后顺序

#### 6.3 通知后联动查询（双保险）

以下4类通知在收到后应立即调用查询接口二次确认：

| 通知类型 | 联动查询接口 |
|---------|------------|
| 协议代扣合约变更通知 | 协议代扣授权状态查询接口（`arrangement.info.query`） |
| 授权代付合约变更通知 | 查询当前授权协议内容接口（`protocol.arr.query`） |
| 授权代付签约结果通知 | 查询申请单状态接口（`protocol.sign.apply.query`） |
| 付款结果通知 | 对应的付款结果查询接口 |

### 7. 退保相关常见问题

#### 7.1 退保流程概览

```
扣保通知（网商→平台）
  ↓ 生成扣保记录（状态：已扣保）
退保申请（平台→网商）
  ↓ 获得 OperateNo（操作单号）
退保查询（平台→网商）
  ↓ 返回 SUCCESS / FAIL / DEALING
退保成功通知（网商→平台）
  ↓ 更新扣保记录状态：已退保
```

#### 7.2 退保查询返回 FAIL

**处理**：
- 记录 `Msg` 字段中的失败原因
- 支持重试退保申请
- 检查原扣保记录是否存在、`DepositPayId` 是否正确

#### 7.3 退保查询一直返回 DEALING

**处理**：
- 定时轮询查询接口（建议每30分钟）
- 退保超时（>7天）：人工介入，联系网商处理
- 检查是否有退保成功通知到达（通知可能先到，查询还显示DEALING）

#### 7.4 未收到退保通知

**处理**：
- 退保成功后网商通过同一通知接口（`ant.mybank.bkcloudbatch.stmt.deposit.notify`）通知，`NotifyType=RETURN_PLATFORM_DEPOSIT`
- 定时轮询退保查询接口作为兜底
- 确认通知回调地址是否正确

#### 7.5 扣保与退保配对

**关键规则**：
- 扣保通知时 `RefundDepositOutTradeNo`（退保外部请求单号）为空
- 退保通知时 `RefundDepositOutTradeNo` 填充，与退保申请中的 `OutTradeNo` 一致
- 系统需自行维护扣保与退保的配对关系
- 通过 `DepositPayId` 关联扣保记录和退保记录

#### 7.6 退款扣保单状态流转

```
待发起 → 处理中 → 退保成功
              ↘ 退保失败
```

| 状态 | 来源 | 映射依据 |
|------|------|---------|
| 待发起 | 本地状态 | 扣保通知已接收，退保申请未发起 |
| 处理中 | 退保单映射 | 退保查询接口返回 Status=DEALING |
| 退保成功 | 退保单映射 | 退保查询接口返回 Status=SUCCESS |
| 退保失败 | 退保单映射 | 退保查询接口返回 Status=FAIL |

#### 7.7 退保幂等控制

- 扣保通知可能重复：通过 `DepositPayId` 去重
- 退保通知可能重复：通过 `RefundDepositOutTradeNo` 去重
- 退保申请需保证 `OutTradeNo` 唯一性

### 8. 接口调用常见错误码

#### 8.1 通用返回码

| 返回码 | 中文描述 | 处理方式 |
|--------|---------|---------|
| `0000` | 处理成功 | 正常处理 |
| `S` | 成功（ResultStatus） | 接口调用成功，需进一步检查业务状态 |
| `F` | 失败（ResultStatus） | 查看 ResultCode 和 ResultMsg 获取失败原因 |
| `U` | 处理中（ResultStatus） | 异步处理中，等待通知或主动查询 |
| `BKMP_ILLEGAL_PARAM` | 参数非法 | 检查接口入参是否合法 |

#### 8.2 接口字段必填标识

| 标识 | 含义 | 说明 |

#### 8.3 常见入参错误

| 错误场景 | 错误描述 | 解决方式 |
|---------|---------|---------|
| OutTradeNo 重复 | 幂等键重复导致返回缓存结果 | 确保每次请求使用唯一的 OutTradeNo |
| IsvOrgId 错误 | 平台机构号不正确 | 确认使用正确值：`202211000000000004381` |
| MerchantId 格式错误 | 网商商户号应为2268开头 | 确认使用的是网商商户号而非外部商户号 |

| AuthDetail base64编码错误 | 授权事项需json格式base64编码 | 确认先json序列化再base64编码 |

#### 8.4 常见业务错误场景

| 场景 | 可能原因 | 排查方向 |
|------|---------|---------|
| 补单创单失败 | OutTradeNo 重复 / PayeeMerchantId 不正确 | 检查外部交易号唯一性和收款商户号 |
| 创建批次失败 | 文件未上传到SFTP / BatchNo重复 | 先确认文件上传成功再创建批次 |
| 代扣失败 | 商户余额不足 / 合约未生效 / 商户被限权 | 先查合约状态，再查余额，最后查限权 |
| 代付失败 | 收款方银行卡不支持 / 超出限额 / 合约失效 | 确认银行卡类型、限额和合约状态 |
| 提现失败 | 未完成短信验证 / 余额不足 / 超单笔限额 | 确认验证流程和余额充足 |
| 退保申请失败 | DepositPayId 不正确 / 扣保记录不存在 | 核对扣保通知中的原始数据 |

## 业务规则

### 1. 异步通知重试策略

网商异步通知的重试间隔为：`1,3,5,10,15,30,60,300,600,900,1800,3600` 秒。如果平台未返回 `ResultStatus=S`，网商会持续重试。平台需在 **10秒内** 返回成功响应。

### 2. 设计原则："通知+查询"双保险

- 关键业务状态变更优先依赖网商被动通知
- 同时提供手动查询兜底
- 通知可能丢失或延迟，定时查询可兜底
- 建议收到通知后立即调用查询接口二次确认

### 3. 各业务终态状态汇总

| 业务 | 终态成功 | 终态失败 | 中间态 |
|------|---------|---------|--------|
| 预入驻 | RegisterStatus=1 | RegisterStatus=2 | RegisterStatus=0（审核中） |
| 激活/注册 | RegisterStatus=1 | RegisterStatus=2 | RegisterStatus=0（注册中） |
| 清算批次 | SUCCESS | FAIL | 清算中 |
| 退保 | Status=SUCCESS | Status=FAIL | Status=DEALING |
| 提现 | Status=SUCCESS | Status=FAIL | Status=DEALING |
| 协议代扣划扣 | WithholdStatus=成功 | WithholdStatus=失败 | WithholdStatus=处理中 |
| 授权代付 | Status=成功 | Status=失败 | Status=处理中 |
| 解冻 | UnfreezeStatus=成功 | UnfreezeStatus=失败 | - |

## 注意事项

1. **ResultStatus=S 不等于业务成功**：必须检查业务状态字段（如 RegisterStatus、BatchStatus、Status 等）确认最终结果。

2. **金额单位**：网商接口中金额单位通常为"分"（如 TotalAmount、Amount），但部分接口为"元"（如退保接口中的 Amount），需仔细核对接口文档。

3. **通知乱序问题**：商户限权通知可能乱序到达，使用 `PunishDate`（管控时间）判断最新状态。

4. **文件有效期**：
   - PART_SUCCESS 时的校验失败文件 `VerifyFailDataFileUrl` 默认三天有效期
   - 资金账单下载链接5分钟有效期（base64编码）

5. **清算失败订单重提**：必须沿用原始商户映射，不得动态更新商户映射，避免重提时商户信息变更导致对账失败。

6. **提现确认前不扣款**：提现申请阶段不会扣减余额，确认接口调用成功后才扣款。

7. **测试环境验证码**：提现短信验证码填 `888888`。

8. **最大单笔提现金额**：1000万。

9. **当日支付当日补单**：当日支付机构交易须在当日完成补单交易创建，账单 D+1 后发送至网商。

## 常见误区

1. **误区：收到通知就可以不做查询了**
   - 实际：通知可能丢失或延迟，系统设计原则是"通知+查询"双保险。特别是4类关键通知（协议代扣合约变更、授权代付合约变更、签约结果、付款结果），收到通知后应立即联动查询确认。

2. **误区：ResultStatus=S 代表业务处理成功**
   - 实际：ResultStatus=S 只代表接口调用成功。例如入驻查询返回 S 但 RegisterStatus 可能是审核中（0）或失败（2），清算查询返回 S 但 BatchStatus 可能是 FAIL。

3. **误区：退保查询返回 FAIL 就无法挽回了**
   - 实际：退保失败后可记录失败原因（Msg字段）并重试。退保超时7天可人工介入联系网商处理。

4. **误区：清算批次 PART_SUCCESS 时成功的那部分没问题**
   - 实际：PART_SUCCESS 表示部分成功，必须下载校验文件（VerifyFailDataFileUrl）逐笔确认每笔订单的具体结果，不能假设未失败的订单都已成功。

5. **误区：重复通知可以忽略不处理**
   - 实际：即使是重复通知也需要返回 `ResultStatus=S`，否则网商会持续重试。正确的做法是持久化通知消息并做幂等处理——已处理过的直接返回成功。

6. **误区：退款需要调用补单创单接口上送退款单**
   - 实际：网商不需要退款单上送，系统中直接将退款订单状态变更为"已上送"即可，不调接口。退款通过扣保/退保机制处理。

7. **误区：协议代扣扣款失败就是余额不足**
   - 实际：代扣失败可能是合约未生效（检查 ArrangementStatus）、商户已解约、商户被限权等多种原因，需查看 FailReason 字段确定具体原因。

8. **误区：清算批次可以并行处理**
   - 实际：前一批次完结后才能处理下一批（串行处理），不支持并行清算。

9. **误区：退保金额一定等于扣保金额**
   - 实际：退保通知中的 Amount（退保金额）可能与扣保金额一致或小于扣保金额。
