# 退款与退保

> 所属系统：Basalt OS 清算分账系统
> 关联接口：3.56 / 3.57 / 3.58
> 最后更新：2026-06-08

---

## 概述

退款与退保是清算分账系统中的异常资金处理链路。当退款发生时，若商户结算单轧差为负且商户可用余额不足，网商银行会先从平台保证金户（8888 开头，又称平台结算户）扣除保证金垫付（即"扣保"），待商户后续有足够余额时再由平台发起退保申请将保证金退回。

扣保与退保基于**商户维度**汇总，不按订单维度。系统需自行维护扣保与退保的配对关系。

---

## 操作步骤

### 扣保退保 4 步完整流程

#### 步骤 1：扣保通知（网商 → 平台）

清算批次结束后，如果发生扣保，网商通过扣保通知接口异步通知平台。

- **接口**：3.56 补单保证金扣除&退回通知接口（`ant.mybank.bkcloudfunds.billpay.platform.deposit.notify`）
- **通知类型**：`PAY_FOR_STMT_NOT_ENOUGH`（商户结算资金不足，扣保证金）
- **核心参数**：

| 字段 | 说明 | 示例 |
|------|------|------|
| DepositPayId | 扣保单编号（网商生成的扣保流水号） | 2026060500001 |
| BatchNo | 清算批次编号 | 20260605001 |
| MerchantId | 网商商户号（2268 开头） | 226801000000000101099 |
| Amount | 扣保金额（单位：元） | 1500.00 |
| NotifyType | 通知类型 | PAY_FOR_STMT_NOT_ENOUGH |
| RefundDepositOutTradeNo | 退保外部请求单号 | **扣保通知时为空** |

- **平台处理**：收到通知后生成扣保记录（状态：已扣保），记录 DepositPayId。

#### 步骤 2：发起退保申请（平台 → 网商）

平台收到扣保通知后，**在商户余额充足时**调用退保申请接口。

- **接口**：3.57 结算单退保申请接口（`ant.mybank.bkcloudbatch.stmt.deposit.return`）
- **入参**：

| 字段 | 说明 | 示例 |
|------|------|------|
| BatchNo | 批次号（与扣保通知一致） | 20260605001 |
| TaskType | 任务类型，固定值 | STMT_AVAILABLE_SUM_SHARE |
| MerchantId | 网商商户 ID（2268 开头） | 226801000000000101099 |
| OutTradeNo | 外部交易号（平台生成，格式：{年月日}{自增ID}） | 2026060500001 |
| DepositPayId | 扣保单据号（与扣保通知一致） | 2026060500001 |

- **回参**：返回 OperateNo（网商操作单号），用于后续查询退保结果。

#### 步骤 3：查询退保结果（平台 → 网商）

平台调用查询接口确认退保处理结果。

- **接口**：3.58 退保结果查询接口（`ant.mybank.bkcloudbatch.stmt.deposit.return.query`）
- **入参**：

| 字段 | 说明 | 示例 |
|------|------|------|
| OutTradeNo | 外部交易号（与退保申请一致） | 2026060500001 |
| OperateNo | 网商操作单号（退保申请返回的操作单号） | 2026060500001 |

- **出参**：

| 字段 | 说明 | 枚举值 |
|------|------|--------|
| Status | 退保处理状态 | SUCCESS（成功）/ FAIL（失败）/ DEALING（处理中） |
| Msg | 失败时的错误描述 | 成功时为空 |

#### 步骤 4：退保通知（网商 → 平台）

退保成功后，网商通过同一通知接口通知平台退保成功。

- **接口**：3.56 补单保证金扣除&退回通知接口（`ant.mybank.bkcloudfunds.billpay.platform.deposit.notify`）
- **通知类型**：`RETURN_PLATFORM_DEPOSIT`（商户结算单退保&明细单退保）
- **核心参数**：

| 字段 | 说明 | 关联关系 | 示例 |
|------|------|---------|------|
| RefundDepositOutTradeNo | 退保外部请求单号 | 对应退保申请中的 OutTradeNo | 2026060500001 |
| NotifyType | 通知类型 | 固定值 | RETURN_PLATFORM_DEPOSIT |
| DepositPayId | 扣保单据号 | 与扣保通知一致 | 2026060500001 |
| Amount | 退保金额（单位：元） | 与扣保金额一致或小于 | 1500.00 |

- **平台处理**：收到退保成功通知后，更新扣保记录状态为"已退保"，退保流程结束。

### 流程时序

```
网商银行                    平台系统                    数据库
   |                          |                          |
   |  1. 扣保通知              |                          |
   |  PAY_FOR_STMT_NOT_ENOUGH |                          |
   |  (DepositPayId, Amount)  |                          |
   +------------------------->|                          |
   |                          |  2. 生成扣保记录          |
   |                          |  (状态:已扣保)           |
   |                          +------------------------->|
   |                          |                          |
   |  3. 退保申请              |                          |
   |  (OutTradeNo, DepositPayId)                        |
   |<-------------------------|                          |
   |                          |                          |
   |  4. 返回操作单号           |                          |
   |  (OperateNo)             |                          |
   +------------------------->|                          |
   |                          |                          |
   |  5. 查询退保结果          |                          |
   |  (OutTradeNo, OperateNo) |                          |
   |<-------------------------|                          |
   |                          |                          |
   |  6. 返回状态              |                          |
   |  (SUCCESS/FAIL/DEALING)  |                          |
   +------------------------->|                          |
   |                          |                          |
   |  7. 退保成功通知          |                          |
   |  RETURN_PLATFORM_DEPOSIT |                          |
   |  (RefundDepositOutTradeNo)                         |
   +------------------------->|  8. 更新扣保记录          |
   |                          |  (状态:已退保)           |
   |                          +------------------------->|
```

## 业务规则

### deposit.notify 5 种通知类型详解

| 类型 | Type 枚举值 | 说明 | 处理优先级 | 备注 |
|------|-----------|------|-----------|------|
| **类型 1** | `PAY_FOR_STMT_NOT_ENOUGH` | 商户结算资金不足，扣保证金 | **P0（核心场景）** | 必须处理。记录 DepositPayId，后续用于退保 |
| **类型 2** | `PAY_FOR_MERCHANT_FREEZE` | 退款单分账未退回——明细单扣保 | - | **已废弃，不需要处理** |
| **类型 3** | `PAY_FOR_TAIL_DIFF` | 三方支付退款尾差，扣保补充 | P2（极少触发） | 兜底场景，不需要处理 |
| **类型 4** | `REFUND_BY_TAIL_DIFF` | 三方支付退款尾差，多余部分退还保证金 | - | **无法使用，需人工处理** |
| **类型 5** | `RETURN_PLATFORM_DEPOSIT` | 商户结算单退保&明细单退保 | **P0（核心场景）** | 必须处理。退保成功通知，退保流程结束 |

实际系统只需处理 3 种：类型 1（扣保）、类型 3（尾差兜底）、类型 5（退保结果）。其中类型 1 和类型 5 为必须处理的核心场景。

### 退保外部请求单号（RefundDepositOutTradeNo）

这是区分扣保通知与退保通知的关键字段：

- **扣保通知（类型 1）**：`RefundDepositOutTradeNo` 字段**为空**
- **退保通知（类型 5）**：`RefundDepositOutTradeNo` 字段**有值**，值为平台发起退保申请时传入的 `OutTradeNo`

系统通过该字段关联扣保记录和退保记录，实现配对管理：

```
扣保通知 -> 生成扣保记录（状态：已扣保）
    |
后续入账 -> 触发退保判断
    |
退保通知 -> 根据 RefundDepositOutTradeNo 关联扣保记录
    |
更新扣保记录状态：已退保
```

### 退款规则：部分退款 vs 全额退款

- **退款订单不需要上送网商**：系统中直接将退款订单状态变更为"已上送"即可，不调用补单创单接口
- **退款单基于支付订单发生**：退款金额不得超过原支付金额
- **最大可退金额** = 付款金额 - 已退款金额 - 退款在途金额
- **退款导致扣保**：当退款使商户结算单轧差为负且可用余额不足时，网商自动从平台 8888 保证金户扣保垫付

### 平台垫资机制

- **扣保资金来源**：平台保证金户（8888 开头的结算户）
- **垫资本质**：平台先行垫付退款，后续等待商户入账后通过退保回收
- **风险点**：如果商户长期无入账，垫资将持续挂账，需定期清理超期挂账（建议 >30 天告警）

### 退保触发时机

**正确做法**：等商户可用余额充足后再发起退保申请。

**已知问题（PRD P0 级）**：收到扣保通知后不应立即发起退保。如果商户余额仍然不足，退保申请会失败。系统应在商户后续有清算入账、余额充足后再触发退保。

**实现建议**：

1. 收到扣保通知后记录扣保记录，状态为"待发起"
2. 监听商户入账事件（清算成功通知 / 来账通知）
3. 检查商户可用余额是否 >= 扣保金额
4. 余额充足时自动或手动发起退保申请

### 退款扣保单状态机

```
待发起 -> 处理中 -> 退保成功
                \-> 退保失败
```

| 状态 | 说明 | 来源 |
|------|------|------|
| 待发起 | 扣保通知已收到，平台尚未发起退保申请 | 本地状态 |
| 处理中 | 退保申请已发起，网商正在处理 | 退保查询接口返回 Status=DEALING |
| 退保成功 | 网商退保成功 | 退保查询接口返回 Status=SUCCESS |
| 退保失败 | 网商退保失败 | 退保查询接口返回 Status=FAIL |

### 退款流水号规则

- **协议代扣流水号**：前缀 **N**（体系内子户间划转）
- **授权代付流水号**：前缀 **W**（体系外付款）
- **退保外部交易号（OutTradeNo）**：格式为 `{年月日}{自增ID}`，如 2026060500001，由平台生成并保证唯一性
- **批量付款批次号**：`00120260525`（3 位当日自增 + YYYYMMDD）
- **批量付款单号**：`001_00120260528`（单内序号 + 批次号）
- **单笔付款单号**：`001_50120260528`（单内序号 + 501 起始的自增 + YYYYMMDD）

## 注意事项

### 幂等控制

- 扣保通知可能重复到达，需通过 `DepositPayId` 去重
- 退保通知可能重复到达，需通过 `RefundDepositOutTradeNo` 去重
- 退保申请需保证 `OutTradeNo` 唯一性

### 异常处理

- **退保查询返回 FAIL**：记录失败原因（Msg 字段），支持重试
- **未收到退保通知**：定时轮询查询接口（建议每 30 分钟一次）
- **退保超时（>7 天）**：人工介入，联系网商银行处理

### 扣保与退保配对管理

- 通过 `DepositPayId` 关联扣保记录和退保记录
- 退保成功后，`RefundDepositOutTradeNo` 与退保申请的 `OutTradeNo` 一致
- 系统需维护扣保 -> 退保的完整链路，确保每笔扣保都有对应的退保记录

### 批次完结通知中的扣保信息

批次完结通知接口（3.16 / `ant.mybank.bkcloudfunds.billpay.batch.finish.notify`）中包含扣保统计字段：

- `IsDepositPay`：是否发生结算单扣保（Y/N）
- `DepositPayCnt`：发生扣保的商户数量

可通过这两个字段快速判断当前批次是否需要进行退保处理。

### 平台结算户余额监控

- 平台 8888 保证金户余额不足时，扣保垫付可能失败
- 需设置余额预警线，监控水位
- 扣保垫付挂账超过 30 天需每周清理

## 常见误区

### 误区 1：收到扣保通知后立即发起退保

**错误**：收到扣保通知后立刻调用退保申请接口。

**正确**：应等商户可用余额充足后再发起退保。立即退保在商户余额不足时会失败，这是 PRD 中已标记的 P0 级已知问题。

### 误区 2：扣保通知的 RefundDepositOutTradeNo 有值

**错误**：期望扣保通知（类型 1）中包含退保外部请求单号。

**正确**：仅退保通知（类型 5）的 `RefundDepositOutTradeNo` 有值，扣保通知（类型 1）该字段为空。

### 误区 3：所有 5 种通知类型都需要处理

**错误**：对 deposit.notify 的全部 5 种类型都编写处理逻辑。

**正确**：只需处理类型 1（PAY_FOR_STMT_NOT_ENOUGH，扣保）和类型 5（RETURN_PLATFORM_DEPOSIT，退保结果）。类型 2 已废弃，类型 3 极少触发可作兜底，类型 4 无法使用需人工处理。

### 误区 4：退款单需要上送网商

**错误**：退款订单也调用补单创单接口上送网商。

**正确**：退款订单不需要上送，系统中直接将状态变更为"已上送"即可，不调接口。只有支付订单需要调用补单创单接口。

### 误区 5：退保金额一定等于扣保金额

**错误**：假设退保通知中的 Amount 一定等于扣保通知中的 Amount。

**正确**：退保金额可能与扣保金额一致，也可能小于扣保金额（部分退保场景），需以退保通知实际返回的金额为准。

### 误区 6：退保成功后不需要再关注

**错误**：收到退保查询接口返回 SUCCESS 后就认为流程结束。

**正确**：应以网商的退保通知（类型 5 / RETURN_PLATFORM_DEPOSIT）为最终确认依据，退保查询接口仅作为兜底查询手段。收到退保通知后更新扣保记录状态为"已退保"，流程才算真正结束。
