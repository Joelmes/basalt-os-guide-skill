# 角色权限

> 适用范围：Basalt OS 清算分账系统
> 来源文档：Basalt OS 核心业务逻辑与实现方案 v6.0、网商银行集中收银分账接口整理 v3.1

---

## 概述

Basalt OS 是一个多角色协作的清算分账平台，涉及品牌方（平台方）、区域公司、门店商户三类组织层级，以及管理员、财务人员、操作员、审核员四类系统角色。不同角色在资金归集、清算批次提交、合约管理、付款提现等流程中承担不同职责，拥有不同的操作范围和数据可见性。

> **说明**：源文档中角色权限并非独立章节，以下内容从各业务流程中提取。标注"【推断】"的部分是基于业务流程逻辑合理推断的操作边界，实际系统中可能有所调整。

---

## 操作步骤

### 1. 系统角色定义

| 角色 | 职责定位 | 典型使用场景 |
|------|---------|------------|
| **管理员** | 系统配置与基础数据管理 | 组织架构创建、门店管理、POS绑定、注册码生成 |
| **财务人员（品牌方财务）** | 资金操作与对账 | 资金归集（手动汇款）、清算操作、对账、手续费管理 |
| **操作员** | 清算数据准备 | 订单上送、清算文件生成与上传、批次创建（仅保存） |
| **审核员** | 清算批次审批与提交 | 审核清算批次、执行"保存并上报" |
| **商户** | 通过网商小程序完成入驻与资金管理 | 入驻激活、协议签约、提现、解约 |

### 2. 各角色操作范围

#### 2.1 管理员

| 操作 | 说明 | 数据来源 |
|------|------|---------|
| 创建总部公司 | 编码格式 CAREBD + 3位自增，如 CAREBD001 | 系统生成 |
| 创建区域公司 | 编码格式 CAREREG + 区域编号 + 3位自增 | 系统生成 |
| 创建/管理门店 | 编码格式 CARE + 省份缩写 + YYYYMMDD + 3位自增 | 三联系统拉取 + 手动完善 |
| 完善门店资料 | 补充门店所属区域等关联信息 | 手动维护 |
| 生成注册码 | 为门店主体"新增注册码"，发起商户入驻流程 | 系统生成 |
| POS机绑定门店 | 手动将POS机绑定到对应门店（或Excel批量导入） | 财务手工维护 |
| 商户信息管理 | 调用商户信息修改接口修改联系人、结算卡等 | 接口 `ant.mybank.merchantprod.merch.updateMerchant` |

> **注意**：所有编码一经生成不可删除、不可修改。POS-商户绑定目前依赖手工Excel维护，存在数据一致性隐患。

#### 2.2 财务人员（品牌方财务）

| 操作 | 说明 | 接口/方式 |
|------|------|----------|
| 资金归集（手动汇款） | 每日从华夏银行汇款到网商9903专户 | 银行转账 |
| 确认到账 | 查看来账通知或手动拉取来账记录 | 被动通知 + 定时查询 + 手动查询 |

## 业务规则

### 1. 清算批次两步审批设计

清算批次采用**两步提交机制**，分离"数据准备"和"正式提交"两个环节：

**第一步：仅保存（操作员）**
- 基于"已上传"的清算文件创建批次
- 仅保存在本地系统，不调用网商接口
- 批次状态为"待上报"
- 清算文件状态同步变为"待上报"

**第二步：保存并上报（审核员）**
- 调用网商创建批次接口 `ant.mybank.bkcloudbatch.batch.create`
- 成功后获得网商批次实例号 `BatchInstanceId`
- 批次状态变为"清算中"

**设计原因**：允许操作人员创建批次后先检查确认，再由审核员正式提交清算。前一批次完结后才能处理下一批次（串行处理）。

**状态流转**：
```
待上报 ──上报──→ 清算中 → 清算成功 / 部分成功 / 清算失败
  ↓       ↓
可作废   上报失败（可重试或作废）
```

### 2. 商户限权管理

商户限权由网商银行发起，通过异步通知接口 `ant.mybank.merchantprod.merch.control.notify` 通知平台。管控中的商户会交易失败，建议接入此接口可在交易前提前收到通知并处理。

#### 限权通知类型（ControlNotifyType）

| 值 | 含义 | 说明 |
|---|------|------|
| **1** | 管控（限权） | 商户被限制操作，交易将失败 |
| **2** | 解除管控 | 商户限权被解除，恢复正常 |
| **3** | 弱管控（弱限权） | 部分功能受限，非完全限制 |
| **4** | 延迟管控（T+N管控） | 延迟N天后执行管控 |

#### 管控类别（ControlCategory）

| 值 | 含义 |
|---|------|
| 1 | 身份类 |
| 2 | 司法类 |
| 3 | 风险类 |

#### 管控策略（ControlStrategy）

| 值 | 含义 |
|---|------|
| 1 | 限制出金 |
| 2 | 限制入金 |
| 3 | 限制出入金 |
| 4 | 不限制 |

#### 通知关键字段

| 字段 | 说明 |
|------|------|
| MerchantIds | 商户号列表（逗号分隔，超过50个会被截断） |
| OutMerchantIds | 外部商户号列表（逗号分隔，超过50个会被截断） |
| ChangeReason | 限制原因（非对客） |
| CanUpgrade | true=可引导升级解除，false=不允许 |
| UpgradeUrl | 中心化升级链接（需支付宝登录态） |
| ControlOrderNo | 管控单号 |
| PunishDate | 管控时间（yyyyMMddHHmmss），可用于处理通知乱序 |

> **建议**：此接口为建议必接。限权通知的幂等处理需注意，通知可能重复发送。

### 3. 协议签约权限差异

| 维度 | 协议代扣（余额划转） | 授权代付 |
|------|-------------------|---------|
| 签约发起方 | 商户（小程序主动签约） | 平台（主动发起签约申请） |
| 签约确认方 | 商户自行完成 | 商户通过确认链接完成 |
| 解约发起方 | 商户（需平台审核） | 商户（无需审核） |
| 平台审核权 | 有（审核解约申请） | 无（商户解约直接生效） |
| 平台取消权 | 无 | 有（可取消待确认的申请单） |

### 4. 付款操作权限【推断】

| 操作 | 总部公司 | 区域公司 | 门店 |
|------|---------|---------|------|
| 协议代扣（批量/单笔） | 可发起 | 可发起 | 可发起 |
| 授权代付（批量/单笔） | 可发起 | 可发起 | 可发起 |

> **说明**：0.5.0版本待开发清单中明确"总部新增付款入口"、"区域公司新增付款入口"、"门店新增付款入口"，说明三个层级均有付款权限。

## 注意事项

1. **编码不可变**：所有组织编码（总部公司、区域公司、门店、外部商户号）一经生成不可删除、不可修改。

2. **POS绑定手动维护**：当前POS-商户绑定依赖财务手工Excel维护，无系统级校验机制，存在数据一致性隐患。未绑定门店的POS机，其账单无法参与上送。

3. **两步审批不可跳过**：清算批次的两步提交必须按顺序执行，操作员创建后审核员才能上报，不能跳步。

4. **商户限权通知乱序**：通知中的 `PunishDate`（管控时间）可用于处理通知乱序到达的情况，应以最新管控时间为准。

5. **限权通知截断**：单次通知中 MerchantIds 和 OutMerchantIds 超过50个会被截断，需关注是否有遗漏。

6. **提现需短信验证**：提现申请后需商户完成短信验证才能确认，测试环境验证码为 `888888`。

7. **保证金户不可感知**：商户保证金户（9903开头）由商户自行通过小程序开通，品牌方无查询接口、无被动通知接口。

8. **串行处理**：前一批次完结后才能处理下一批，不支持并行清算。

## 常见误区

1. **误区：操作员可以直接上报清算批次**
   - 实际：操作员只能"仅保存"创建批次，必须由审核员执行"保存并上报"才能正式提交清算。

2. **误区：协议代扣的签约由平台发起**
   - 实际：协议代扣（余额划转）的签约由商户在小程序主动完成，平台侧无主动签约接口。只有授权代付的签约才由平台发起。

3. **误区：商户解约协议代扣后立即失效**
   - 实际：商户发起协议代扣解约后，状态变为"解约中待平台确认"（INVALID_TO_BE_CONFIRM），需要平台审核通过后合约才失效。平台可同意或拒绝解约。

4. **误区：授权代付解约也需要平台审核**
   - 实际：授权代付的解约由商户发起后直接生效，平台无需审核，网商通过 `ant.mybank.bkcloudfunds.protocol.sign.notify` 通知平台即可。

5. **误区：品牌方可以查询商户保证金户余额**
   - 实际：品牌方对商户保证金户（9903开头）无查询接口、无被动通知接口，完全无法感知其开通和余额变化。

6. **误区：所有区域的门店都可以归属多个区域**
   - 实际：每个门店只能归属1个区域。但一个门店可以挂靠多个营业执照，每个营业执照可用于多个商户入驻。

7. **误区：清算文件上传后可以直接上报**
   - 实际：清算文件上传后状态为"已上传"，需要先创建批次（第一步：仅保存），文件状态变为"待上报"，然后审核员才能上报。

8. **误区：商户限权通知中 ControlNotifyType=4 表示立即管控**
   - 实际：ControlNotifyType=4 表示"延迟管控"（T+N管控），是延迟N天后才执行管控，不是立即执行。
