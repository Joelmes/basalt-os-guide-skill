# 外部系统同步

## 概述

Basalt OS 清算分账系统共有 **10处** 需要从外部系统拉取或同步数据，涉及3个外部系统：

- **网商银行**（API接口）：7处被动通知 + 联动查询
- **三联ERP**（API接口）：1处定时拉取 + 手动拉取
- **收钱吧**（FTP推送）：1处外部推送

核心设计原则是 **"通知+查询"双保险**：关键业务状态变更优先依赖网商被动通知，同时提供手动查询兜底；订单数据则依赖定时+手动两种拉取方式。

## 操作步骤

### 10个同步点完整列表

#### 同步点1：品牌专户来账明细（网商银行API）

- **定时拉取**：每天凌晨3点调用来账明细查询接口（`ant.mybank.bkcloudfunds.payment.query`）查询 D-1 全量入账明细
- **被动通知**：网商通过渠道结算入账通知接口（`ant.mybank.bkcloudfunds.settle.notify`）在入账/出账时实时通知
- **手动拉取**：用户点击「更新」按钮查询当日全量入账明细；点击「拉取来账记录」查询截至操作时间的增量来账
- **数据格式**：JSON
- **幂等要求**：通过 OrderNo（来账订单号）去重

#### 同步点2：品牌专户余额（网商银行API）

- **按需查询**：调用子户外标余额查询接口（`ant.mybank.bkcloudfunds.balance.query`），在品牌专户页面展示待清算金额
- **触发条件**：用户主动查看品牌专户页面时触发
- **数据格式**：JSON，Amount 单位为分
- **幂等要求**：查询接口可重复调用，无需幂等处理

#### 同步点3：三联ERP清算订单（三联ERP API）

- **定时拉取**：每天凌晨3点调用三联`零售单查询(open.saleorder.query)`拉取 D-1 全量订单
- **手动拉取**：用户点击「拉取订单」按钮，拉取截至操作时间的增量订单
- **处理方式**：异步处理
- **数据格式**：接口JSON
- **触发条件**：定时任务 + 用户手动触发
- **幂等要求**：需根据订单号去重，避免重复入库
- **已知PRD缺失项**：三联ERP订单通道目前为预留设计，暂未启用；PRD中未完整定义三联订单字段映射和异常处理规则

#### 同步点4：收钱吧清算订单（收钱吧FTP推送）

- **收钱吧推送**：收钱吧每天自动推送一次对账单 Excel 文件到可柔 FTP 服务器（D-1 全部账单）
- **三联辅助**：该模块复用同步点3中三联凌晨3点的定时拉取数据，用于匹配门店名称等补充字段
- **手动拉取**：无（不支持手动拉取）
- **数据格式**：Excel对账单
- **触发条件**：收钱吧每日自动推送
- **幂等要求**：根据账单日期和账单ID去重
- **前置条件**：POS机已绑定门店，门店已关联商户

#### 同步点5：清算批次结果（网商银行API）

- **被动通知**：网商通过补单批次完结通知接口（`ant.mybank.bkcloudbatch.batch.finish.notify`）在批次清算完成时回调通知
- **手动更新**：用户点击「更新批次状态」按钮，调用批次查询接口（`ant.mybank.bkcloudbatch.batch.query`）查询指定或全部"清算中"批次的最新状态
- **数据格式**：JSON
- **触发条件**：清算批次上报后等待网商处理完成
- **幂等要求**：通过 BatchInstanceId 去重
- **三种结果**：SUCCESS（全部成功）、PART_SUCCESS（部分成功，需下载校验文件确认）、FAIL（全部失败）

#### 同步点6：退款扣保与退保结果（网商银行API）

- **被动通知（扣保）**：网商通过补单保证金扣除&退回通知接口（`ant.mybank.bkcloudbatch.stmt.deposit.notify`）在扣保发生时通知（Type=`PAY_FOR_STMT_NOT_ENOUGH`）
- **被动通知（退保成功）**：退保成功后通过同一通知接口通知（Type=`RETURN_PLATFORM_DEPOSIT`）
- **手动更新**：用户点击「更新退保状态」调用退保结果查询接口（`ant.mybank.bkcloudbatch.stmt.deposit.return.query`）查询"处理中"状态的退保单
- **数据格式**：JSON
- **触发条件**：清算批次结束后如有扣保则触发
- **幂等要求**：扣保通知通过 DepositPayId 去重；退保通知通过 RefundDepositOutTradeNo 去重

#### 同步点7：协议代扣合约变更（网商银行API）

- **被动通知**：商户签约/解约时，网商通过授权/解约通知接口（`ant.mybank.merchantprod.merchant.arrangement.info.notify`）实时通知平台
- **联动查询**：收到通知后立即调用协议代扣授权状态查询接口（`ant.mybank.merchantprod.merchant.arrangement.info.query`）再确认一次合约最新状态
- **数据格式**：JSON
- **触发条件**：商户在网商云资金小程序发起签约或解约时
- **幂等要求**：通过 ArrangementNo（合约号）去重

#### 同步点8：授权代付合约变更（网商银行API）

- **被动通知（合约变更）**：商户签约/解约时，网商通过授权合同变更通知接口（`ant.mybank.bkcloudfunds.protocol.sign.notify`）实时通知平台
- **联动查询（合约内容）**：收到通知后立即调用查询当前授权协议内容接口（`ant.mybank.bkcloudfunds.protocol.arr.query`）查询合约最新内容
- **被动通知（异步签约结果）**：网商通过协议签约异步申请结果通知接口（`ant.mybank.bkcloudfunds.protocol.sign.apply.notify`）返回签约/修改结果
- **联动查询（申请单状态）**：收到签约/修改通知后立即调查询申请单目前状态接口（`ant.mybank.bkcloudfunds.protocol.sign.apply.query`）确认最新状态
- **数据格式**：JSON
- **触发条件**：平台发起签约申请后商户确认/修改/解约时
- **幂等要求**：通过 AgreementNo（合约号）去重

#### 同步点9：付款结果（网商银行API）

- **被动通知（协议代扣）**：协议授权代扣通知结果接口（`ant.mybank.bkcloudfunds.protocol.withhold.result.notify`）
- **被动通知（授权代付）**：授权代付结果通知接口（`ant.mybank.bkcloudfunds.trade.operate.result.notify`）
- **联动查询**：收到通知后再用查询接口查一次付款单数据
- **手动更新**：用户点击「更新付款状态」按钮，分别调用对应查询接口查询全部"处理中"状态的付款单
- **数据格式**：JSON
- **触发条件**：发起协议代扣或授权代付后等待结果
- **幂等要求**：通过付款单号去重

#### 同步点10：商户入驻状态（网商银行API）

- **被动通知（小程序入驻）**：小程序入驻商户注册通知接口（`ant.mybank.merchantprod.merch.notify`）
- **被动通知（预入驻激活）**：预入驻激活结果通知接口
- **按需查询**：通过预入驻结果查询、激活结果查询、小程序入驻结果查询等接口轮询状态
- **数据格式**：JSON
- **触发条件**：商户在小程序完成入驻/激活操作后
- **幂等要求**：通过 OutMerchantId（外部商户号）去重

## 业务规则

### "通知+查询"双保险机制详解

系统的核心同步原则是"通知+查询"双保险，具体体现在以下三个层面：

**第一层：被动通知为主**
- 网商银行通过异步通知接口（Webhook）在关键业务状态变更时实时推送通知到平台
- 通知失败重试策略：1, 3, 5, 10, 15, 30, 60, 300, 600, 900, 1800, 3600（单位：秒）
- 平台需在10秒内返回成功响应（ResultStatus=S），否则视为通知失败

**第二层：联动查询为辅**
- 收到通知后立即调用对应的查询接口再确认一次最新状态
- 目的是防止通知数据与实际状态不一致（如通知传输过程中数据丢失或篡改）
- 4处同步点采用了联动查询策略（同步点7、8、9涉及）

**第三层：手动查询兜底**
- 所有关键同步点都提供了用户手动触发的查询入口
- 用户点击按钮主动拉取/更新最新状态
- 用于处理通知完全丢失的极端情况

**定时拉取兜底（针对订单数据）**：
- 每天凌晨3点自动拉取 D-1 全量数据
- 与通知数据互补，确保数据完整性

### 同步方式统计

| 同步方式 | 出现次数 | 说明 |
|---------|---------|------|
| 定时自动拉取 | 2处 | 均为每天凌晨3点执行（来账明细、三联订单） |
| 外部推送到本方 | 1处 | 收钱吧每日推送Excel到FTP |
| 被动接收通知（Webhook） | 7处 | 网商各类通知接口回调 |
| 通知后联动查询 | 4处 | 收到通知后立即调查询接口二次确认 |
| 用户手动触发 | 6处 | 用户点击按钮主动拉取/更新 |

### 幂等处理要求

**所有通知类接口都需要做幂等处理**，使用唯一业务单号去重：

| 同步点 | 幂等键 | 说明 |
|-------|--------|------|
| 来账明细 | OrderNo | 来账订单号 |
| 清算批次结果 | BatchInstanceId | 网商批次实例号 |
| 扣保通知 | DepositPayId | 扣保单据号 |
| 退保通知 | RefundDepositOutTradeNo | 退保外部请求单号 |
| 协议代扣合约 | ArrangementNo | 合约号 |
| 授权代付合约 | AgreementNo | 合约号 |
| 付款结果 | 付款单号 | 系统生成的付款单号 |
| 商户入驻 | OutMerchantId | 外部商户号 |
| 补单创单 | OutTradeNo | 外部交易流水号 |
| 批次创建 | IsvOrgId+BatchNo+AcquireId | 三者组合做幂等 |

## 注意事项

1. **通知可能丢失或延迟**：网商异步通知不保证100%送达，必须实现查询兜底机制
2. **通知可能重复**：同一事件可能收到多次通知（重试机制导致），所有通知处理逻辑必须做幂等
3. **联动查询不可省略**：收到通知后的联动查询不是多余的，是确保数据一致性的关键步骤
4. **定时任务失败需告警**：凌晨3点的定时拉取任务如果失败，需要有告警机制并支持手动重试
5. **FTP推送无手动拉取能力**：收钱吧订单只支持FTP被动推送，不支持手动拉取，如果推送失败需要联系收钱吧重新推送
6. **三联ERP通道暂未启用**：三联订单通道为预留设计，目前实际业务中订单全部来自收钱吧
7. **手动触发覆盖不全**：并非所有同步点都有手动触发入口，部分同步点仅依赖被动通知

## 常见误区

1. **误以为通知接口保证100%送达**：异步通知可能因网络、超时等原因丢失，必须有查询兜底
2. **误以为收到通知就不需要查询了**：系统设计原则是"通知+查询"双保险，收到通知后仍需联动查询确认
3. **误以为收钱吧订单可以手动拉取**：收钱吧只支持FTP每日推送，没有手动拉取能力
4. **误以为三联ERP订单已在使用**：三联订单通道目前为预留设计，暂未启用，实际业务订单全部来自收钱吧
5. **误以为所有通知都需要平台审核**：只有协议代扣的解约需要平台审核，授权代付的解约无需审核
6. **误以为幂等只需要在通知接收端做**：幂等处理需要在所有数据写入环节做，包括通知接收、查询结果处理、定时拉取入库等
7. **误以为三联ERP的PRD已完整覆盖**：三联ERP同步的PRD存在已知缺失项，包括字段映射不完整和异常处理规则未定义
