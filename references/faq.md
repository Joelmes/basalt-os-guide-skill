## 一、主体、入驻与注册码

### Q1. 哪些类型的主体可以入驻？不同主体的基本信息怎么产生？

系统内可入驻的主体分四类：**门店、供应商、总部公司、区域公司**。

- **总部公司、区域公司**：由平台在 Basalt OS「清算分账」模块内**手动创建**。填写公司信息保存后，系统按规则生成公司编码，基本信息在本系统产生并维护。
- **门店、供应商**：基础数据从**三联系统同步**而来，本系统不手工新建：

  - 门店：基础数据来自三联（门店名称、类型、状态等），通过「拉取新门店」增量同步、「更新门店信息」拉最新；门店编码由系统生成。如需修改门店基础信息，须到三联系统改后再同步。
  - 供应商：从三联系统同步供应商档案，通过「拉取新供应商」「更新供应商信息」同步；供应商编码由系统生成。
  - 门店/供应商的**经营主体与法人信息**（营业执照、统一社会信用代码、法人、法人身份证、收款账户等）在本系统补充维护，不同步回三联。

> 要点：创建/同步主体后，还须生成「商户注册码」并交由法人完成扫码入驻，才会产生真正的网商商户。

### Q2. 商户入驻需要哪些资料？有几种入驻方式，分别怎么操作？

**入驻方式**：当前全部为**「小程序入驻」**一种。后续版本计划增加「预入驻 + 激活」方式（预入驻方式会多出入驻状态：预入驻审核中 / 待激活 / 预入驻失败）。

**操作流程（平台侧 + 法人侧）**：

1. 平台运营在 Basalt OS 中创建主体（门店/供应商/总部公司/区域公司）并生成商户注册码（见 Q4）。
2. 平台将注册码的**二维码图片或注册链接**通过钉钉分发给对应法人。
3. 法人使用**本人支付宝**扫描二维码（或在钉钉内打开链接），进入网商银行云资金小程序，按指引自行完成入驻：

   - 选择企业证件类型（选「营业执照」）；上传营业执照图片（若已在网商认证过会自动带出）；
   - 确认/上传法人身份证正反面；
   - 填写经营者/联系人/电话、与营业执照一致的地址；
   - 绑定收款账户：可先绑支付宝，提现时再绑银行卡；若绑银行卡，只能选法人/负责人**本人**银行卡。

**法人侧需准备的资料**：营业执照（图片 + 公司名称）、法人身份证正反面、联系人/电话、经营地址。

> ⚠️ **重要提醒：商户入驻只能由营业执照上的法人使用本人支付宝操作，不可由他人代办。** 平台侧负责建主体、生成注册码、分发注册码；实际入驻由法人完成。

### Q3. 如何筛选未生成注册码的门店，批量生成注册码，并批量下载未入驻门店的注册码？

这三步横跨**两个页面**（生成在「门店列表」，下载在「门店商户列表」），关键切换点如下：

**第一步：筛出还没生成注册码的门店**

1. 进入 **门店列表**（清算分账 → 门店 → 门店列表）。
2. 筛选区「商户注册码」选 **「无注册码」**，得到全部未生成码的门店。
3. 勾选这些门店（或全选筛选结果）。

**第二步：批量生成注册码**

- 点击批量操作区 **「（批量）生成注册码」**。
- 弹窗两选项：

  - **仅生成**：只落库，不下载；
  - **生成并下载**：异步生成并**立即打包下载**二维码 .zip（仅含本次新生成的门店）。

**第三步：批量下载"还没入驻"的注册码**

- 去 **门店商户列表**（清算分账 → 门店 → 门店商户）：

  1. 筛选「入驻状态」选 **「未入驻」**（覆盖所有尚未扫码入驻的商户，含历史遗留）；
  2. 全选筛选结果 → 点击 **「下载注册码」** → 确认 → 系统异步打包 .zip 下载。

**两条路线怎么选**

- 只想下载本次新生成的 → 第二步直接选「生成并下载」，一步到位。
- 想下载"所有未入驻"（含以前生成但还没扫的）→ 第二步选「仅生成」，第三步在门店商户列表按「未入驻」全量下载。

**注意事项**

1. 门店商户的入驻状态不会自动刷新，下载前建议先点「更新入驻状态」确认，避免误判。
2. 注册码不可撤销 / 不可删 / 不可改，分发前必须核对门店信息。
3. 一个门店可对应多个注册码；批量生成时即使已有码也会再新增，可能造成重复商户。第一步用「无注册码」筛选已自然排除有码门店，不会重复。
4. 生成 / 下载均为异步，完成后浏览器自动下载 .zip，文件名形如「门店名称+外部商户号」。

### Q4. 商户注册码跟外部商户号是什么关系？外部商户号跟门店/公司/供应商编码有什么联系？怎么获取、下载、使用？

**注册码 ↔ 外部商户号**：一个商户注册码对应一个**外部商户号**（即网商商户号 `MerchantId`）。每新增一个注册码，就生成一个独立外部商户号；因此"一个主体多个注册码"等价于"一个主体多个外部商户号"。

**外部商户号 ↔ 主体编码**：外部商户号基于主体编码生成，规则为 `{主体编码}_{2位自增编号}`。例如门店编码 `CAREBJ20260415001` 对应外部商户号 `CAREBJ20260415001_01`、`..._02`；总部公司/区域公司/供应商同理使用各自的公司编码 / 供应商编码作前缀。

**怎么获取（生成）注册码**：在对应主体列表（门店列表 / 总部公司列表 / 区域公司列表 / 供应商列表）点击 **「新增注册码」**，生成即落库（不可撤销），并在对应商户列表顶部新增一条商户数据。一个主体可新增多个注册码（无上限）。

**怎么下载注册码**：在对应**商户列表**（门店商户 / 总部公司商户 / 区域公司商户 / 供应商商户）勾选商户 → 点击 **「下载注册码」** → 系统异步打包 .zip（图片名形如「主体名称 + 外部商户号」）。此外，第二步批量生成时的「生成并下载」选项也会直接打包本次新生成的码。

**注册码怎么用**：注册码本质是一个入驻入口。把注册码对应的**二维码图片或注册链接**通过钉钉分发给法人，法人用本人支付宝扫码进入网商银行云资金小程序完成入驻（见 Q2）。

### Q5. 我想找某个门店/公司/供应商有哪些关联商户，怎么找？怎么查商户入驻状态？

**找关联商户（某主体下的全部商户）**：

- 在该主体列表（门店/总部公司/区域公司/供应商列表）中，外部商户号后会有超链接 **「等 x 个商户 >」**，点击即跳转至对应商户列表（门店商户 / 总部公司商户 / 区域公司商户 / 供应商商户），并自动带入该主体编码筛选，展示其关联的全部商户。
- 也可直接在商户列表用「主体名称 / 主体编码 / 商户简称 / 外部商户号」筛选定位。

**查商户入驻状态**：

- 在商户列表查看「入驻状态」列；系统目前支持两种入驻路径，完整状态流转如下（以实际展示状态为准）：

| 入驻状态 | 含义 | 关注建议 |
|-|-|-|
| 未入驻 | 注册码已生成，法人尚未发起入驻 | 正常初始态；需推进法人入驻，否则无法走清算 |
| 预入驻审核中 | 平台运营已发起预入驻，网商正在审核 | 待办：点「更新入驻状态」查结果 |
| 待激活 | 预入驻审核通过，等待法人激活 | 待办：通知法人完成激活 |
| 激活中 | 法人已激活（小程序驻商户路径由法人扫码直接进入到此态），网商正在异步处理 | 待办：点「更新入驻状态」查结果（不自动刷新） |
| 激活成功 | 入驻成功，商户可用 | 终态，正常使用 |
| 预入驻失败 | 预入驻审核未通过（仅预入驻路径） | **重点关注**：状态后有提示符号可看完整失败原因，修正后可"重新激活"回到预入驻审核中 |
| 激活失败 | 激活未通过 | **重点关注**：状态后有提示符号可看完整失败原因；修正后"重新激活"回到对应节点（小程序驻商户路径回激活成功，预入驻路径回激活中） |

- **两种入驻路径**（按截图状态流转图）：

  1. **小程序驻商户**：未入驻 → 激活中 → 激活成功（失败可"重新激活"回激活成功）。
  2. **预入驻 + 激活**：平台运营发起预入驻 → 未入驻 → 预入驻审核中 → 待激活 → 激活中 → 激活成功（任一失败可"重新激活"回到对应节点继续推进）。
- 入驻状态非实时，勾选商户后点 **「更新入驻状态」** 调用网商接口拉取最新。
- **需重点关注的状态**：「预入驻失败」「激活失败」（须查明原因并重新激活）；「预入驻审核中」「待激活」「激活中」（须手动刷新确认结果）；长期「未入驻」会阻塞清算，应推动法人完成。

### Q6. 一个门店可以入驻多个商户码（注册码）吗？

**可以。** 一个门店（主体）可新增多个注册码，每个注册码对应一个独立的外部商户号（生成规则见 Q4）。

- 外部商户号形如 `CAREBJ20260415001_01`、`CAREBJ20260415001_02`。
- 当主体关联多个商户时，列表默认展示最早创建的商户信息，并显示「等 x 个商户」链接（见 Q5）。

---

### Q7. 一个营业执照可以入驻多个商户吗？

**可以。** 网商银行侧允许**一个营业执照认证多个商户**。门店与商户是 **1 对多**的绑定关系，具体一个门店绑定几个商户、是否多个门店共用同一营业执照下的商户，由**业务侧**决定。

> 绑定关系由业务决定，系统按业务配置落地。注意：一个门店可对应多个注册码/外部商户号（见 Q6），多个商户意味着多套独立子户与对账，运营需关注资金归集与对账的对应关系。

---

### Q8. 注册码生成后可以删除或修改吗？

**不可以。** 注册码一旦新增即不可撤销，不支持删除和修改操作。生成后对应商户数据立即出现在商户列表（初始入驻状态「未入驻」）。请在分发前确认信息准确。

---

### Q9. 商户入驻只能由法人操作吗？

**是。** 商户入驻只能由营业执照上的法人使用本人支付宝操作，不可由他人代办。平台运营（平台侧）负责在 Basalt OS 中创建主体并生成注册码，注册码通过钉钉分发给法人，由法人扫码进入网商银行云资金小程序自行完成入驻。

## 二、订单获取

### Q10. 清算订单从哪里来？收钱吧和拉卡拉有什么区别？

清算订单原始数据来自两家收银机构的对账单：

| 项目 | 收钱吧（安徽） | 拉卡拉（河南） |
|-|-|-|
| 拉取方式 | **自动**：每日 12:00 前推送 D-1 账单至可柔 FTP，系统 14:00 拉取解析 | **手动**：财务从拉卡拉后台下载对账单，在本系统上传 |
| 当前状态 | 实际订单全部来自收钱吧 | 通道为预留设计，尚未大规模启用 |
| 订单唯一标识 | 交易流水号（支付/退款单不同） | 交易流水号（退款单关联原交易流水号） |

- 本系统订单状态从「订单上送」环节开始定义（三联系统未维护订单前置状态）。
- 当前实际订单来源 100% 为收钱吧 FTP 推送；拉卡拉通道与三联 ERP 清算订单通道均尚未大规模上线。

### Q11. 门店收银的钱怎么进入清算体系？

资金进入清算体系的链路（含线下人工步骤）：

```Plaintext
门店 POS 收款
  → 资金落收钱吧/拉卡拉监管户，D+1 结算至个体工商户法人的华夏卡（POS 绑定个体工商户法人的华夏卡）
  → 财务人员手动将资金转入网商平台清算专户（9903 开头）
  → 收钱吧账单同步至系统（D-1 推送）
  → 订单上送 → 生成清算文件 → 批次上报 → 网商清算
```

关键点：**资金归集（华夏银行卡 → 9903 清算专户）是线下人工转账步骤**，系统不自动完成，需财务在批次上报前确保清算专户待清算余额大于清算批次实际金额。

### Q12. POS 机必须绑定商户吗？怎么绑定？是一次性的还是每次都要做？

**必须绑定。** 订单上送的前置条件就是 POS 已绑定商户，未绑商户的订单无法上送。

**绑定操作**（清算分账 → POS 设置）：

- **单个绑定**：在 POS 列表点「绑定」，弹窗中按门店 / 商户筛选目标商户，选中确定。
- **Excel 批量导入**：下载模板，填写 POS 机名称（须与收钱吧对账单**严格一致**）、商户简称、门店名称，上传导入。

**绑定关系**：多对一——一个 POS 只能绑定至一个商户，一个商户可绑定多个 POS。

**是否一次性**：绑定是**一次性的配置关系**，建立后长期有效，后续该 POS 的订单会自动归属所绑商户，无需每次清算重复绑定。系统**暂不提供自助解绑**；如绑定错误确需调整，需联系技术人员处理。

> 注意：门店/商户的绑定关系直接影响清算资金的划转，请仔细核对（模板中已用红色字段标为必填并提示风险）。

### Q13. 订单状态分别是什么意思？平台用户需要关注哪些？异常状态有哪些、怎么处理？

订单共 6 个状态：

| 状态 | 含义 |
|-|-|
| 未上送 | 订单拉取至系统后的初始状态，尚未向网商发起上送 |
| 上送失败 | 调用网商补单创单接口失败（未拿到网商订单号）；退款单上送还需其原支付单已上送成功 |
| 已上送 | 支付单上送成功拿到网商订单号；退款单不调接口，状态直接变更 |
| 清算中 | 清算文件上传成功、批次上报成功，网商处理中 |
| 清算成功 | 批次清算完成，该订单结果为成功 |
| 清算失败 | 批次清算完成，该订单结果为失败（可重新生成清算文件后发起） |

**平台用户需要关注的状态**：

- **未上送 / 上送失败**：待办，需执行「订单上送」。
- **清算中**：需后续「更新批次状态」查看结果（状态不自动刷新）。
- **清算失败**：需重新处理（见 Q33）。
- 已上送（中间态，待生成清算文件）、清算成功（终态，无需操作）正常关注即可。

**异常状态及处理**：

- **上送失败**：重新上送即可；若是退款单失败，先确认其原支付单已上送成功（原支付单未上送则退款单必失败）。**若原支付单尚未上送、退款单先到达，退款单上送会直接失败，需先完成原支付单上送后，再重新上送该退款单。**
- **清算失败**：见 Q33 处理流程。

流转：`未上送 → 上送失败/已上送 → 清算中 → 清算成功/清算失败`

## 三、资金归集与账户

### Q14. 资金归集是系统自动的还是人工的？

**部分自动、部分人工。**

- **自动部分**：门店 POS 收款后，资金先进入收钱吧/拉卡拉监管户，D+1 自动结算至个体工商户法人的华夏卡。
- **人工部分**：资金进入品牌方银行卡后，需**财务人员手动转账**至网商为品牌方开具的平台清算专户（9903 开头）。这一步系统无法代执行，必须在清算批次上报前完成，且确保清算专户待清算余额 ≥ 清算批次实际金额；若余额不足，批次虽能上报但会一直卡在「清算中」（网商持续重试、不会直接失败，见 Q22）。

> 该步骤可提前做（订单上送完成后即可安排转账），不必等到清算文件生成后。

### Q15. 平台清算专户、保证金户、商户子户分别是什么？

| 账户 | 网商编号前缀 | 说明 |
|-|-|-|

### Q17. 平台保证金户（8888）是干什么用的？需要注意什么？

**主要用途**：

1. **网商扣除服务费**：品牌服务费（如商户提现的网商手续费、授权代付的网商手续费）由网商向**品牌方**收取，D+1 出账单，从保证金户扣除（明细见品牌服务费对账文件，D+2 拉取）。
2. **清算时退款扣保垫付**：清算批次中若某商户有退款订单且可用余额不足，网商从保证金户为该商户垫付退款（即扣保，见 Q21）。
3. **服务费扣缴资金回流**：平台根据品牌服务费对账文件，通过「协议代扣」向商户收取手续费（提现手续费由商户承担），扣缴成功后资金进入保证金户。

**注意事项**：

- **余额必须充足**：扣保垫付、网商服务费扣除都从该户出，余额不足会导致扣保/扣费失败。
- 在「品牌服务费」页面关注「未付金额」与「扣缴状态」，及时对"未扣缴/失败"账单发起服务费扣缴（扣缴从商户子户协议代扣至保证金户）。
- **对账文件刷新时间**：品牌服务费对账文件由系统**每日凌晨 3:00 自动拉取前一日数据**。若白天在「品牌服务费」页查看「未付金额」发现未更新，属正常现象，待次日 3:00 自动拉取后刷新，无需手动操作。
- 在「品牌专户」页面定期查看保证金户余额，发现偏低时及时补足。

## 四、清算

### Q18. 一个完整的清算流程是什么？要完成哪些工作？每一步需要注意什么？

完整链路（"POS 收款""资金转入清算专户"为线下步骤，系统只处理信息流）：

```Plaintext
收钱吧/拉卡拉 POS 收款 → 资金归集（财务手动转账到 9903 清算专户）
→ 订单同步至系统 → 订单上送（POS 须已绑商户）
→ 生成清算文件 → 上传清算文件
→ 新建清算批次并上报 → 网商按批次处理 → 返回清算结果
（如有扣保）退保
```

**逐步工作与注意事项**：

1. **资金归集（线下）**：转账至 9903 清算专户，确保待清算余额 ≥ 清算批次实际金额。可提前做；余额不足不会直接失败，而是卡在「清算中」、网商持续重试（见 Q22）。
2. **POS 绑定商户（前置）**：未绑定商户的订单无法上送（见 Q12）。批量绑定时 POS 名称须与对账单严格一致。
3. **订单上送**：筛选「未上送 / 上送失败」订单勾选上送。支付单调网商补单创单接口；退款单不调接口，但需其原支付单已上送成功。
4. **生成清算文件**：勾选「已上送」且批次号为「-」的订单，或「清算失败」订单生成（清算失败订单批次号会被重置为「-」）。
5. **上传清算文件**：上传清算文件与同名标档文件，两文件都成功才算上传成功。
6. **新建批次并上报**：可「仅保存」审核或「保存并上报」正式提交（两步设计见 Q19）。
7. **（如有扣保）退保**：清算后若发生扣保，待商户余额充足时手动发起退保（见 Q21）。

---

### Q19. 清算批次上报为什么要分"仅保存"和"保存并上报"两步？

设计为**防误操作**：

- **仅保存**：批次数据仅在本系统保存（状态「待上报」），不调用网商接口，供操作员审核批次内容。
- **保存并上报**：保存后立刻调用网商创建批次接口正式提交清算。

两步分离让操作员在正式提交前有机会复核，避免误将错误批次上报给网商。

---

### Q20. 每天清算文件上传次数有限制吗？"实际金额"指什么？

**上传次数限制**：每天最多上传 **50 次** 清算文件，请合理规划每批次的订单数量。

**实际金额定义**：每个清算文件中，**全部支付订单的实际金额 − 全部退款订单的实际金额**。

- 另有「交易总金额」= 实际金额 + 渠道手续费；「渠道手续费」= 全部支付订单渠道手续费 − 全部退款订单渠道手续费。
- 每个清算文件（清算批次）的「实际金额」须 **＞ 0**，否则无法清算。

---

### Q21. 什么是扣保？什么是退保？

- **扣保**：清算批次提交后，网商按商户维度聚合清算。若某商户有退款订单且可用余额不足，网商从平台保证金户（8888）为该商户垫付退款，即为扣保。网商通过通知接口逐商户发送扣保通知，系统自动创建扣保单（**通知消息可能重复到达，系统已做幂等处理，运营无需关心重复扣保单**）。
- **退保**：将平台 8888 户被垫付的资金从商户可用余额中归还的过程。扣保单初始状态「待发起」，需运营人员在商户余额充足时手动点击「退保」发起申请；退保成功后台通过通知接口告知平台。

> 扣保/退保状态同样需手动「更新退保状态」刷新（见 Q31）。

---

### Q22. 清算批次一直卡在"清算中"可能是什么原因？

「清算中」本身是**正常的异步处理态**——批次上报后由网商按批次处理，结果通过网商通知/查询接口返回，系统不实时自动刷新。长时间未见结果，按顺序排查：

1. **未手动更新状态**：批次状态不会自动刷新。先在批次管理页点 **「更新批次状态」**（不勾选时系统自动查询列表所有「清算中」批次的最新状态）。多数"卡住"只是没刷新。
2. **网商侧正常处理延迟**：大批次或高峰期处理较慢，更新后仍「清算中」可稍后再次更新。
3. **清算专户待清算资金不足**：若 9903 清算专户待清算余额小于批次实际金额，网商侧会**一直重试、不会直接报失败**，表现为批次清算状态一直停留在「清算中」（即"卡住"）；**没有具体的阈值或重试次数限制**。需在品牌专户页确认 9903 待清算余额是否充足，补足后再观察。
4. **异常未终结**：若更新后变为「清算失败 / 上报失败」，按对应失败原因处理（参数问题等，见 Q18、Q33）。
5. **仍长时间无结果**：排除以上后，可能是网商侧异常，联系**产品或网商客服**人员排查。

> 提示：清算专户待清算余额小于批次实际金额时，网商侧会一直重试（无具体阈值/重试次数限制），不会直接报失败，表现为批次长期卡在「清算中」，需优先核对 9903 余额并补足。

### Q24. 授权代付有哪些业务场景和限额？外部银行卡有什么要求？

授权代付分五个业务子场景，收款方支持银行卡（对公/对私银联借记卡，不支持信用卡）和支付宝（仅个人对私）：

| 子场景 | 场景码 | 单笔限额 | 月度总额 | 收款方账户上限 | 签约生效规则 |
|-|-|-|-|-|-|
| 租金 | RENT | 2.5 万 | 25 万 | 5 | 首次签约：需门店法人签约授权  <br/>信息变更：需门店法人签约确认 |
| 水电物业费 | FEE | 2 万 | 10 万 | 10 | 首次签约：需门店法人签约授权  <br/>信息变更：需门店法人签约确认 |
| 合伙人分润 | PARTNER | 10 万 | 40 万 | 40 | 首次签约：需门店法人签约授权  <br/>信息变更：需门店法人签约确认 |
| 工资代发 | SALARY | 2 万 | 25 万 | 50 | 首次签约：需门店法人签约授权  <br/>后续变更：需门店法人签约确认 |

- **外部银行卡要求**：仅支持银联**借记卡**（对公 / 对私均可），**不支持信用卡**；对公银行卡需填联行号；支付宝仅支持**个人对私**账户，不支持对公。
- **一个银行卡/支付宝账户不能在同一合约的多个子场景中重复出现**（每个账户在一个授权代付合约中只出现一次）。

### Q25. 协议代扣、授权代付是否需要签约？分别怎么签约？在哪里查看签约状态？

**都需要签约。**

**协议代扣（商户签）**：

- 由商户在支付宝云资金小程序中**主动发起**签约，网商通过通知接口将合约号返回平台，合约自动创建（商户:合约 = 1:1）。
- 平台侧查看即可，无法代为发起。

**授权代付（平台发起、商户确认）**：

1. 前置：商户已在网商云资金小程序自行开通**过桥西进户**（见 Q26）。
2. 平台在「签约授权 → 新增授权代付合约」填写表单（授权方商户、子场景与收款方等）并提交。
3. 系统返回**申请编号**和**商户确认链接**：收款方 ≤ 50 走同步接口立即返回；＞ 50 走异步，处理完成后通过通知返回状态与确认链接。
4. 商户通过确认链接完成授权，网商返回合约号，合约创建。

**查看签约状态**：进入 **清算分账 → 签约授权 → 合约列表**，可按「合约类型（协议代扣 / 授权代付）、授权方名称 / 编码、商户简称 / 商户号」筛选；点击某合约「详情」查看合约状态（生效 / 失效等）与最新申请单状态。

### Q26. 授权代付签约需要注意什么？过桥西进户是干什么的、怎么开通？授权代付合约与该商户关联的外部收款账户有什么关系？在哪里查看绑定的银行卡/支付宝？

**授权代付签约注意事项**：

- **必须先开通过桥西进户**：商户未在网商开通过桥西进户，授权代付申请无法成功（见下）。
- 收款方银行卡仅银联借记卡（对公/对私），不支持信用卡；支付宝仅个人对私；一个账户不能在同一合约多子场景重复。
- 五个子场景收款方总数 ≤ 200；至少保留一个子场景有收款方。

**过桥西进户（桥西进户）**：

- **干什么用**：过桥西进户是商户侧用于**授权代付资金过渡**的账户（过桥户）。发起授权代付向外部银行卡/支付宝付款时，资金经由该过桥户转出。
- **怎么开通**：由**商户在网商云资金小程序中自行开通**，品牌方无法代为操作；且**网商不会通知、也无法查询商户是否已开通**。因此在发起授权代付申请前，务必向商户确认已完成开通。
- 合约详情会展示过桥户（户名-账号）；未填写时网商使用商户最新过桥西进户，无值则显示「-」。

**合约与外部收款账户的关系**：

- 授权代付合约与商户、过桥户为 **1:1:1** 关系（一个商户对应一个合约、一个过桥户）。
- 合约绑定了各业务子场景下的**外部收款账户**（即租金/工资等场景的收款方银行卡或支付宝）。这些收款方就是该商户授权代付对外付款的目标账户。

**在哪里查看某商户授权代付合约绑定的银行卡/支付宝**：

- 合约列表筛选该商户 → 点击其「授权代付」合约的 **「详情」** → 查看「业务子场景、子场景限额、子场景收款方」，按租金/水电/合伙人/工资/报销五个子场景展示各自绑定的收款方（户名、账号均**脱敏**展示）。

## 六、提现与资金回款

### Q27. 钱是怎么回到门店的？（提现）

清算完成后，网商按商户维度将资金分账至各商户子户（2268）的可用余额。门店/商户要拿到钱，需主动**提现**：

1. 商户通过 **API 接口**发起提现（注：小程序提现通道将关闭，统一走 API）；
2. 系统调用网商单笔提现申请接口，资金从商户子户（可提现金额）提至结算卡；
3. 商户实际到账金额 = 申请提现金额 − 品牌提现手续费（0.1%）。

即：门店收银 → 归集到 9903 → 清算分账到 2268 子户 → 商户提现（子户可提现金额 → 结算卡）。

---

### Q28. 门店提现怎么操作？手续费怎么收？

- **商户侧**：通过 API 接口发起提现，从子户可用（可提现）余额提至绑定的结算银行卡。
- **品牌提现手续费（0.1%）**：提现申请时由平台直接从申请金额中扣除，商户看到的是全额到账（实际到账 = 申请金额 − 0.1%）。**当前默认费率为 0.1%，未来将支持可配置。**
- **网商提现手续费（0.05%）**：商户提现成功后，网商 **D+1** 从平台 8888 保证金户扣除前一日全部商户的提现手续费；平台 **D+2** 通过品牌服务费对账文件获取明细并回填。

---

### Q29. 商户子户出金有额度限制吗？

- 子户间划转（体系内，协议代扣）：**无限制**。
- 子户对外付款分两类：①**授权代付**受分场景月度限额约束（如合伙人分润月限 40 万，各场景独立，详见 Q24）；②**提现**不属授权代付五场景限额体系，按提现规则处理（见 Q27/Q28）。
- 额度不足时，可在同一营业执照下注册多个商户号以分散额度（见 Q7）。

## 七、跨模块综合问题

### Q30. 商户入驻成功后，必须马上走清算流程吗？

**不是，清算不是入驻后的自动触发动作。**

入驻成功仅表示商户在网商侧开户完成。要走清算，必须同时具备：

1. 门店已有收银订单（依赖 POS 绑定商户 + 收钱吧推送订单）；
2. 资金已归集至 9903 清算专户；
3. 订单已完成上送、生成清算文件并上报。

此外，若需做分账/付款（协议代扣、授权代付），还需先完成签约授权（见 Q25）。因此入驻成功是前置条件，但清算、分账均按需、按订单触发，不强制即时执行。

---

### Q31. 入驻状态、清算批次状态为什么不会自动刷新？需要手动更新吗？

**是，部分状态需手动触发刷新。**

- **入驻状态**：网商异步返回，系统不实时拉取。运营需在商户列表勾选商户后点「更新入驻状态」调用网商接口查询最新状态；入驻失败的状态后会有提示符号，点击可看完整失败原因。
- **清算批次状态**：勾选批次点「更新批次状态」查询；不勾选直接点时，系统自动查询列表中所有「清算中」批次的最新状态（见 Q22）。
- **扣保/退保状态**：同理需点「更新退保状态」。

### Q32. 一个主体同时有"多个商户"时，列表怎么展示？

当主体（门店/公司/供应商）关联多个商户时：

- 主体列表默认展示最早创建的商户信息；
- 外部商户号后用超链接显示「等 x 个商户」，点击跳转至对应商户列表并带入主体编码筛选，展示该主体关联的全部商户（见 Q5）。

### Q33. 清算失败或部分成功怎么办？

- **全部失败（清算失败）**：批次内订单全部失败。此时订单会**退回到「已上送」状态（不需要重复上送）**，只需**重新生成清算文件 → 上报清算批次**以发起清算。
- **部分成功**：据网商开发反馈，现阶段**一般不会出现**部分成功（只会出现全部成功或全部失败）。若实际出现，请联系**产品或网商客服**人员处理（网商会返回失败信息下载地址，三天有效）。

> 清算文件**已作废**后可重新生成；**清算中及之后**状态无法撤回，只能等结果。

### Q34. 有哪些工作是我应该每天检查的？哪些需要特别注意？

建议每日例行检查以下四项（均可在对应页面操作）：

1. **检查清算批次状态**：批次管理页点「更新批次状态」，确认无长期「清算中」未结、无「清算失败 / 上报失败」遗留（处理见 Q22、Q33）。
2. **检查退款扣保情况**：退款扣保列表查看有无「待发起」扣保单，确认对应商户余额充足后及时「退保」，避免平台保证金户被长期占用（见 Q21）。
3. **检查服务费扣除情况**：品牌服务费页面查看有无「未扣缴 / 失败」账单，及时对商户发起服务费扣缴（从商户子户协议代扣至保证金户，见 Q17）。
4. **检查平台保证金户余额是否充足**：品牌专户页面查看 8888 保证金户余额，确保足以支撑扣保垫付与服务费扣除；余额偏低及时补足。

**额外建议检查**：订单上送情况（有无「未上送 / 上送失败」堆积）、清算专户待清算余额（上报前须 > 清算批次实际金额）。
