# Basalt OS 商户入驻参考手册

## 概述

商户入驻是 Basalt OS 清算分账系统的前置环节。商户入驻前需先在系统中建立门店数据（来源于三联系统），门店就绪后才能发起入驻。系统支持两种入驻路径：预入驻+激活（路径A）和小程序入驻（路径B，主要方式）。入驻成功后商户获得网商分配的网商商户号（MerchantId），后续所有接口调用均依赖此商户号。

---

## 操作步骤

### 前置准备：门店数据建立

1. 从三联系统拉取门店信息
2. 在本系统中完善门店资料（补充门店所属区域等）
3. 为门店对应的主体"新增注册码"，商户初始状态为"未入驻"

### 路径A：预入驻 + 激活（两步流程）

**第一步：预入驻**

1. 平台调用商户预入驻接口（3.1）：`ant.mybank.bkmerchantprod.merch.applet.pre.register`
   - 上传商户信息（身份证、营业执照、联系人、结算卡等）
   - 获得 OrderNo（申请单号）
   - 状态变为"预入驻审核中"
2. 通过预入驻结果查询接口（3.3）轮询：`ant.mybank.merchantprod.merch.register.query`
   - 或等待预入驻结果异步通知接口（3.2）回调
3. 审核通过后状态变为"待激活"

**第二步：激活**

1. 平台按规范生成激活链接/二维码，发放给商户
   - 线上环境链接格式：`alipays://platformapi/startapp?appId=2021001187624111&nbupdate=syncforce&query=outMchId%3D{outMchId}%26isv%3D{ISV_ORG_ID}%26app%3Dbkclfs%26t%3DACTIVATE`
   - 测试环境需额外添加 `&uid%3D{uid}` 参数
2. 商户打开链接，在网商云资金小程序中完成信息认证和激活
3. 通过激活结果通知接口（3.4）接收结果，或通过激活结果查询接口（3.5）主动查询：`ant.mybank.merchantprod.merch.applet.register.query`
4. 最终状态为"激活成功"或"激活失败"

### 路径B：小程序入驻（主要方式）

1. 商户扫描注册码，直接在网商小程序完成入驻
2. 通过小程序入驻商户注册通知接口（3.6）接收回调：`ant.mybank.merchantprod.merch.notify`
3. 或通过小程序入驻结果查询接口（3.7）查询：`ant.mybank.merchantprod.merch.applet.register.query`
4. 最终状态为"激活成功"或"激活失败"

### POS 机绑定

POS 机绑定是收银资金清算的前置步骤：

1. 当出现新的 POS 机时，在系统中手动将该 POS 机绑定到对应门店
2. 绑定方式：手动绑定或 Excel 批量导入
3. 绑定后，该 POS 机下的所有账单在清算时按关联关系分配到所绑定门店的商户子户

**数据链路：** 收钱吧账单 → 关联POS机 → POS机绑定门店 → 门店关联商户 → 清算到商户子户

## 业务规则

### 入驻状态流转

```
未入驻 → 预入驻审核中 → 待激活 → 激活中 → 激活成功
                    ↘ 预入驻失败            ↘ 激活失败
```

### 入驻结果状态码

| 状态码 | 含义 | 说明 |
|--------|------|------|
| 0 | 处理中 | 入驻申请正在审核或激活正在处理 |
| 1 | 成功 | 入驻/激活成功，商户获得网商商户号 |
| 2 | 失败 | 入驻/激活失败，需查看失败原因 |

### 商户限权状态

| 状态码 | 含义 | 说明 |
|--------|------|------|
| 1 | 限权 | 商户被完全限制，交易将失败 |
| 2 | 解除 | 限权已解除，恢复正常 |
| 3 | 弱限 | 部分功能受限 |
| 4 | 延迟 | 延迟处理状态 |

限权通知通过接口（3.30）异步推送：`ant.mybank.merchantprod.merch.control.notify`。管控中的商户会交易失败，建议必接此接口以便在交易前提前收到通知并处理。

### 预入驻接口核心入参

| 参数 | 必填 | 说明 |
|------|------|------|
| IsvOrgId | M | 外部机构号（网商分配） |
| OutTradeNo | ME | 外部交易号（唯一） |
| OutMerchantId | ME | 外部商户号 |
| MerchantType | M | 商户类型：01-自然人，02-个体工商户，03-企业商户 |
| MerchantName | M | 商户名称 |
| DealType | M | 经营类型：01-实体，02-网络，03-兼营 |
| CertPhotoA | M | 身份证正面图片 |
| CertPhotoB | M | 身份证反面图片 |
| LicensePhoto | C | 营业执照图片（个人无需） |
| MerchantDetail | M | 商户详情（JSON base64 编码） |
| BankCardParam | C | 结算卡详情（JSON base64 编码） |

### 编码规则

| 层级 | 格式 | 示例 |
|------|------|------|
| 外部商户号 | 主体编码 + _ + 2位自增 | CAREBJ20260415001_01 |

所有编码一经生成不可删除、不可修改。

## 注意事项

1. **门店数据先行**：商户入驻前必须先在系统中建立门店数据，门店数据来源于三联系统，需完善区域归属等信息后才能生成注册码。
2. **激活链接生成**：激活链接为前端生成（非 API 接口），需替换 `{outMchId}` 和 `{ISV_ORG_ID}`，测试环境额外添加 uid 参数。
3. **预入驻阶段绑卡限制**：预入驻阶段绑卡只支持企业商户；结算到银行卡时 BankCardNo 必填，结算到支付宝时 LogonId 必填。
4. **企业联行号**：企业商户必须上送正确的联行号（ContactLine）。
5. **幂等性**：OutTradeNo 同一交易号被视为同一笔交易，需保证唯一性。
6. **异步通知频率**：网商异步通知按递增间隔发送：1, 3, 5, 10, 15, 30, 60, 300, 600, 900, 1800, 3600 秒。
7. **未绑定 POS 的账单**：未绑定门店的 POS 机，其账单无法参与上送，会在上送时被系统过滤。
8. **商户号获取**：商户激活成功后获得网商分配的 MerchantId（2268 开头），系统需同步存储 MerchantId 与外部商户号的绑定关系。

## 常见误区

1. **误区：预入驻和小程序入驻是同一种方式** — 这是两条独立的入驻路径。预入驻+激活（路径A）由平台先上传商户信息，商户后激活；小程序入驻（路径B）由商户直接扫码完成全部流程。
2. **误区：入驻成功后就能立即清算** — 入驻成功后商户获得 MerchantId，但清算还需要完成 POS 绑定门店、门店关联商户等前置步骤。
3. **误区：状态码 0 表示失败** — 状态码 0 表示"处理中"，1 才是成功，2 是失败。
4. **误区：限权状态只有限权和解除两种** — 实际有4种状态：限权(1)、解除(2)、弱限(3)、延迟(4)，弱限和延迟状态也需要系统处理。
5. **误区：一个门店只能有一个商户** — 一个门店可挂靠多个营业执照，每个营业执照可用于多个网商侧商户的入驻，门店—营业执照—商户是一对多对多关系。
6. **误区：POS 机自动绑定门店** — POS 机绑定门店需要手动操作或 Excel 批量导入，依赖财务手工维护，无系统级自动匹配。
