# 清算流程

## 概述

清算流程是 Basalt OS 的核心业务链路，负责将品牌方汇入的资金通过网商银行体系划拨到各商户子户。完整流程包含 12 个步骤，从订单上送一直到提现和对账，所有步骤必须严格按顺序执行，不能跳步。

核心设计原则是"通知+查询"双保险：关键业务状态变更优先依赖网商被动通知，同时提供手动查询兜底；订单数据则依赖定时+手动两种拉取方式。

---

## 操作步骤

### 12 步完整清算流程

```
01 平台入驻
    ↓
02 商户入驻
    ↓
03 资金归集（品牌方手动汇款 → 网商平台清算专户）
    ↓
04 销售收银（订单/账单拉取）
    ↓
05 订单上送
    ↓
06 创建清算文件并上传 SFTP
    ↓
07 创建批次并上报
    ↓
08 接收清算结果
    ↓
09 退款扣保与退保
    ↓
10 分账（协议代扣 / 授权代付）
    ↓
11 提现
    ↓
12 对账
```

### 步骤 03：资金归集

品牌方财务人员每日手动从华夏银行汇款到网商 9903 专户，系统通过三种方式感知到账：

1. **被动通知**：渠道结算入账通知接口（`ant.mybank.bkcloudfunds.settle.notify`）实时推送
2. **定时查询**：每天凌晨 3 点调用来账明细查询接口（`ant.mybank.bkcloudfunds.payment.query`）查 D-1 全量
3. **手动操作**：用户点击「更新」或「拉取来账记录」按钮，调用子户外标余额查询接口（`ant.mybank.bkcloudfunds.balance.query`）确认余额

完整的日清算时序：D 日品牌方手动汇款 → 确认到账 → 上送订单 → 生成清算文件并上传 SFTP → 创建清算批次并上报 → 网商执行清算 → 接收清算结果。

### 步骤 04：销售收银（订单获取）

系统有两个订单来源：

| 来源 | 获取方式 | 数据格式 | 同步频率 |
|------|---------|---------|---------|
| 收钱吧 | 收钱吧推送到可柔 FTP 服务器 | Excel 对账单 | 每天推送一次 |
| 三联 ERP | 调用 API `open.saleorder.query` | 接口 JSON | 每天凌晨 3 点自动 + 手动拉取 |

> **备注**：目前实际业务中订单全部来自收钱吧，三联 ERP 订单通道为预留设计，暂未启用。

收钱吧的账单关联到 POS 机而非门店，因此需要一个前置步骤：**POS 机绑定门店**。当出现新的 POS 机时，需要在系统中手动将 POS 机绑定到对应的门店。绑定完成后，该 POS 机下的所有账单在清算时就会按照关联关系分配到所绑定门店的商户子户中。

```
收钱吧账单 → 关联 POS 机 → POS 机绑定门店 → 门店关联商户 → 清算到商户子户
```

### 步骤 05：订单上送

**前置条件**：订单所属 POS 机已绑定到门店且门店已关联商户。

- **支付订单**：调用补单创单接口（`ant.mybank.bkcloudfunds.bill.pay`）上送 → 获得网商订单号 OrderNo → 状态变为"已上送"
- **退款订单**：网商不需要退款单上送，系统中直接将状态变更为"已上送"即可，不调接口

**订单状态流转**：

```
未上送 ──上送──→ 已上送 ──关联批次清算──→ 清算中 → 清算成功
   ↘ 上送失败（可重试）                         ↘ 清算失败
```

**重要约束**：当日支付机构交易须在当日完成补单交易创建，账单 D+1 后发送至网商。

### 步骤 06：创建清算文件并上传 SFTP

从"已上送"状态的订单中勾选 → 生成清算文件（文本格式）。

**清算文件命名规则**：`CARE{日期}_{序号}`

- 批次号格式示例：`CARE20260509_01`（品牌名 + 日期 + 2 位当日自增序号）

**SFTP 上传路径**：

```
upload/${isvOrgId}/${acquireId}/${batchNo}/merchant_trade_detail_${batchNo}.txt
```

上传完成后还需上传完结标志文件：`finish_${batchNo}.txt`

**清算文件格式字段**：channel_type, trade_type, batch_no, mybank_order_no, merchant_id, trade_amount 等。

**清算文件上传限制**：每日最多上传 50 次。

**清算文件状态流转**：

```
待上传 → 已上传 → 待上报 → 清算中 → 清算成功/部分成功/清算失败
  ↓         ↓
上传失败   已撤回（从网商网盘删除）
  ↓         ↓
  └── 可作废 ←─┘  （作废后文件内订单恢复为"已上送"）
      ↑
      └── 待上传状态也可直接作废
```

### 步骤 07：创建批次并上报（两步提交机制）

**第一步：仅保存**
- 基于"已上传"的清算文件创建批次
- 仅保存在本地，不调网商接口
- 批次状态为"待上报"
- 清算文件状态同步变为"待上报"

**第二步：保存并上报**
- 调用创建批次接口（`ant.mybank.bkcloudbatch.batch.create`）
- 成功后获得网商批次实例号 `BatchInstanceId`
- 批次状态变为"清算中"

设计两步的原因：允许操作人员创建批次后先检查确认，再正式提交清算。

**创建批次接口核心入参**：

| 参数 | 说明 | 示例 |
|------|------|------|
| BatchNo | 批次号，平台维度下保证唯一 | 仅支持数字、字母、大小写字符 |
| AcquireId | 渠道编号 | 默认传 DEFAULT |
| BatchType | 批次类型 | STMT_AVAILABLE_SUM_SHARE（结算至余额并分账-汇总分账） |
| FileUploadType | 文件上送方式 | SFTP（当前暂仅支持 SFTP） |
| ExtInfo | 扩展字段 | AutoUnfreezeBalance:Y（批次内商户剩余结算资金自动解冻） |

**清算批次状态流转**：

```
待上报 ──上报──→ 清算中 → 清算成功 / 部分成功 / 清算失败
  ↓      ↓
可作废  上报失败（可重试或作废）
```

### 步骤 08：接收清算结果

- **被动通知**：网商通过批次完结通知接口（`ant.mybank.bkcloudbatch.billpay.batch.finish.notify`）回调
- **手动查询**：调用批次查询接口（`ant.mybank.bkcloudbatch.batch.query`）

**三种清算结果**：

| 结果 | 说明 |
|------|------|
| SUCCESS | 全部订单清算成功 |
| PART_SUCCESS | 部分成功，需下载校验文件（VerifyFailDataFileUrl）确认每笔订单结果 |
| FAIL | 全部失败 |

**批次完结通知核心参数**：

| 字段 | 说明 |
|------|------|
| Status | 批次处理状态：SUCCESS / PART_SUCCESS / FAIL |
| VerifyFailDataFileUrl | 校验失败信息下载地址（PART_SUCCESS 时出现，默认三天有效期） |
| IsDepositPay | 是否发生结算单扣保：Y/N |
| DepositPayCnt | 发生结算单扣保的商户数量 |

### 步骤 09：退款扣保与退保

清算批次结束后，如果商户结算资金不足，网商会扣除保证金（扣保），平台需后续发起退保申请。详见退保相关文档。

### 步骤 10：分账

- 协议代扣（体系内子户划转）
- 授权代付（体系外付款）

### 步骤 11：提现

商户发起提现申请 → 短信验证 → 确认提现 → 资金到商户绑定银行卡。

### 步骤 12：对账

通过资金账单下载、对账单查询等接口完成财务对账。

## 业务规则

### 顺序固定

以上 12 步流程必须按顺序执行，不能跳转。前一步骤未完成，后续步骤无法进行。

### 批次与对账文件一一对应

一个批次对应一个对账文件。批次号（BatchNo）在平台维度下必须保证唯一，网商侧基于 `IsvOrgId + BatchNo + AcquireId` 做幂等处理。

### 串行处理

前一批次完结才能处理下一批。不支持多个批次并行清算。

### 扣保与退保链路

上送订单 → 清算 → 上报批次 → 网商扣保 → 平台退保。扣保以商户维度汇总，不按订单维度。

### 清算失败订单重提规则

- 必须沿用原始商户映射
- 不得动态更新商户映射
- 避免重提时商户信息变更导致对账失败

### 补单分账逻辑

补单创单接口（`ant.mybank.bkcloudfunds.bill.pay`）用于预下单补单场景。合作方在支付机构支付成功后调用此接口，生成一笔待入账的云资金订单。平台使用返回的网商订单号（OrderNo）与该笔交易进行关联。

- 针对支付订单明细进行补单创单操作
- 退款订单基于支付订单发生，不进行创单操作
- 接口调用日为该次补单账单批次日

### 资金归集规则（D+1 到账）

品牌方从华夏银行汇款到网商 9903 专户，资金到账确认后方可进行清算操作。资金归集是清算的前置条件。

## 注意事项

1. **POS 机绑定是上送前置条件**：未绑定门店的 POS 机，其账单无法参与上送，会在上送时被系统过滤。
2. **当日交易当日上送**：当日支付机构交易须在当日完成补单交易创建，否则可能影响清算。
3. **SFTP 上传需上传完结标志**：清算文件上传后，必须额外上传 `finish_${batchNo}.txt` 标志文件，否则网商不会处理。
4. **PART_SUCCESS 需下载校验文件**：批次结果为部分成功时，必须下载 VerifyFailDataFileUrl 中的校验文件，逐笔确认每笔订单的清算结果。
5. **清算文件作废后订单恢复**：清算文件作废后，文件内的订单会恢复为"已上送"状态，可以重新勾选生成新的清算文件。
6. **幂等控制**：补单创单接口通过 OutTradeNo 做幂等，同一交易号被视为同一笔交易。创建批次接口基于 `IsvOrgId + BatchNo + AcquireId` 做幂等。
7. **通知重试策略**：批次完结通知失败后会按 1,3,5,10,15,30,60,300,600,900,1800,3600 秒的频率重试。

## 常见误区

1. **误区：三联 ERP 订单通道已启用**
   实际：目前订单全部来自收钱吧，三联 ERP 为预留设计，暂未启用。

2. **误区：退款订单也需要调用补单创单接口上送**
   实际：网商不需要退款单上送，系统中直接将状态变更为"已上送"即可，不调接口。

3. **误区：创建批次时"仅保存"会调用网商接口**
   实际："仅保存"操作仅在本地保存，不调用任何网商接口。只有"保存并上报"才会调用创建批次接口。

4. **误区：清算批次可以并行处理**
   实际：前一批次完结才能处理下一批，不支持并行。

5. **误区：清算文件上传后就可以上报批次**
   实际：清算文件上传后状态变为"已上传"，需要先创建批次（"仅保存"变为"待上报"），再执行"保存并上报"。

6. **误区：收钱吧账单直接关联门店**
   实际：收钱吧账单关联的是 POS 机，需要先在系统中手动将 POS 机绑定到门店，才能参与清算。

7. **误区：清算失败订单重提时可以更新商户映射**
   实际：必须沿用原始商户映射，不得动态更新，避免对账失败。
