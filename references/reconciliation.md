# 对账

## 概述

对账是 Basalt OS 清算分账系统的核心环节，用于确保平台资金流转的准确性。系统涉及两类主要数据源：

- **收钱吧对账表**：收钱吧每天自动推送一次对账单 Excel 文件到可柔 FTP 服务器，包含 D-1 全部账单
- **网商银行回单与账单**：通过网商银行接口获取资金账单、费用对账文件、电子回单等

对账覆盖以下业务场景：
- 订单清算对账（收钱吧订单 vs 网商清算结果）
- 提现手续费对账（网商扣费 vs 向商户追缴）
- 品牌专户来账对账（品牌方汇款 vs 网商入账记录）
- 付款对账（协议代扣/授权代付结果核对）

---

## 操作步骤

### 一、订单数据获取与对账

#### 1. 收钱吧 FTP 文件获取

收钱吧每天自动推送 D-1 全部对账单 Excel 文件到可柔 FTP 服务器。

**数据链路**：
```
收钱吧账单 → 关联POS机 → POS机绑定门店 → 门店关联商户 → 清算到商户子户
```

**前置条件**：
- POS机已在系统中手动绑定到对应门店
- 门店已关联商户（商户已完成入驻）

**注意事项**：
- 未绑定门店的POS机，其账单无法参与上送，会在上送时被系统过滤
- 绑定方式为手动绑定或 Excel 批量导入，由财务手工维护
- 目前实际业务中订单全部来自收钱吧，三联ERP订单通道为预留设计，暂未启用

#### 2. 三联ERP辅助数据

收钱吧订单模块复用三联ERP凌晨3点的定时拉取数据，用于匹配门店名称等补充字段。

#### 3. 订单上送

- **支付订单**：调用补单创单接口（`ant.mybank.bkcloudfunds.bill.pay`）上送，获得网商订单号 OrderNo，状态变为"已上送"
- **退款订单**：网商不需要退款单上送，系统中直接将状态变更为"已上送"即可，不调接口

### 二、提现手续费对账（5步流程）

提现手续费采用 D+1/D+2 后置收取模式，完整对账流程如下：

**第1步：D+2 凌晨3点拉取费用对账文件**
- 调用品牌商费用对账文件申请接口（`ant.mybank.bkmbp.stmt.fee.file.apply`）
- 获取前一日（D+1）网商从平台保证金户扣收的全部手续费账单
- 该文件只包含平台已支付的手续费账单，未支付的账单不会展示

**第2步：筛选提现手续费记录**
- 从对账文件中筛选"提现（小程序模式）"和"提现（API模式）"两类手续费账单

**第3步：与本地提现订单比对，找出新增订单**
- 将费用对账文件中的提现手续费记录与此前提现通知接口获取的提现订单进行比对
- 识别出费用对账文件中存在但本地通知记录中不存在的订单

**第4步：通过查询接口获取商户信息**
- 对新增的提现订单，调用单笔提现查询接口（`ant.mybank.bkcloudfunds.withdraw.query`）查询详情
- 获取提现商户信息，将该提现记录展示在对应商户的提现记录列表中

**第5步：通过协议代扣向商户收取手续费**
- 调用协议授权代扣接口（`ant.mybank.bkcloudfunds.protocol.withhold`）向商户收取手续费
- 协议代扣核心入参：

| 参数 | 值 | 说明 |
|------|-----|------|
| PayerId | 提现商户的网商商户号 | 付款方商户号（2268 开头） |
| PayerType | MERCHANT | 付款方类型：商户 |
| PayeeId | 平台 8888 开头的保证金户（结算户） | 收款方 ID（平台结算户） |
| PayeeType | PLATFORM | 收款方类型：平台 |

### 三、品牌专户来账对账

系统通过三种方式感知品牌方资金到账：

1. **被动通知**：渠道结算入账通知接口（`ant.mybank.bkcloudfunds.settle.notify`）实时推送
2. **定时查询**：每天凌晨3点调用来账明细查询接口（`ant.mybank.bkcloudfunds.payment.query`）查 D-1 全量
3. **手动操作**：用户点击「更新」或「拉取来账记录」按钮，调用子户外标余额查询接口（`ant.mybank.bkcloudfunds.balance.query`）确认余额

## 业务规则

### 对账接口清单

| 接口名称 | 接口代码 | 用途 |
|---------|---------|------|
| 商户子户余额查询 | `ant.mybank.bkcloudfunds.subaccount.balance.query` | 查询商户子户余额 |
| 子户外标余额查询 | `ant.mybank.bkcloudfunds.balance.query` | 查询平台清算专户余额 |
| 账户场景余额查询 | `ant.mybank.bkcloudfunds.merchant.scene.balance.query` | 查询场景余额 |
| 资金账单下载申请 | `ant.mybank.bkmbp.stmt.fund.stmt.apply` | 下载资金账单 |
| 渠道结算入账通知 | `ant.mybank.bkcloudfunds.settle.notify` | 入账通知 |
| 来账明细查询 | `ant.mybank.bkcloudfunds.payment.query` | 查询来账明细 |
| 品牌商费用对账文件申请 | `ant.mybank.bkmbp.stmt.fee.file.apply` | 拉取品牌方费用对账文件 |
| 单笔提现查询 | `ant.mybank.bkcloudfunds.withdraw.query` | 查询提现订单详情 |
| 对账单查询 | `ant.mybank.bkcloudfunds.recon.query` | 查询并获取对账单文件 |

### 回单下载接口

电子回单作为记账凭证，获取流程如下：

1. 调用电子回单申请接口（`ant.mybank.bkcloudfunds.electronicreceipt.apply`）申请单个回单
2. 或调用电子回单批量申请接口（`ant.mybank.bkcloudfunds.electronicreceipt.batchapply`）批量申请
3. 等待电子回单申请结果通知接口（`ant.mybank.bkcloudfunds.electronicreceipt.notify`）异步通知
4. 或主动调用电子回单查询结果接口（`ant.mybank.bkcloudfunds.electronicreceipt.query`）查询
5. 获取回单文件 URL 后下载电子回单

### 资金账单下载

- 调用资金账单下载申请接口（`ant.mybank.bkmbp.stmt.fund.stmt.apply`）
- 传入账单日期和类型
- 返回下载链接（5分钟有效期，base64编码）
- 文件格式为 txt，逗号分隔
- 只能申请昨日及之前的账单，当日账单需次日申请

### 清算失败订单重提规则

- 必须沿用原始商户映射
- 不得动态更新商户映射
- 避免重提时商户信息变更导致对账失败

### 批次与对账文件对应关系

- 一个清算批次对应一个对账文件
- 批次号命名规则：`CARE20260509_01`（品牌名 + 日期 + 2位当日自增）
- 前一批次完结才能处理下一批（串行处理）

## 注意事项

1. **提现通知不含手续费字段**：提现结果通知接口本身不包含手续费金额，必须通过 D+2 的费用对账文件获取
2. **手续费由商户承担但提现时不感知**：商户提现时全额到账，手续费由平台先行垫付（D+1），次日向商户追缴（D+2）
3. **协议代扣失败需人工介入**：如果协议代扣收取手续费时商户余额不足，代扣会失败，需人工介入处理
4. **费用对账文件仅含已支付账单**：品牌商费用对账文件只包含平台已支付的手续费账单，未支付的不会展示
5. **POS-商户绑定是前置条件**：未绑定门店的POS机账单无法参与清算上送
6. **POS-商户绑定依赖手工维护**：当前绑定方式为 Excel 手工维护，存在数据一致性隐患，无系统级校验机制
7. **入账通知可能丢失**：渠道结算入账通知可能因网络等原因丢失，需通过来账明细查询接口定时校对兜底

## 常见误区

1. **误以为提现手续费从提现金额中直接扣除**：实际不是。商户提现金额全额到商户提现卡（银行卡），网商不直接从提现金额中扣手续费，而是 D+1 从平台8888保证金户收取，D+2 平台再通过协议代扣向商户追缴
2. **误以为收钱吧账单可以直接匹配门店**：收钱吧的账单关联的是POS机，而非门店。需要先在系统中将POS机绑定到门店，才能完成匹配
3. **误以为费用对账文件包含所有账单**：费用对账文件只包含平台已支付的手续费账单，如果8888保证金户余额不足导致扣费失败，该笔账单不会出现在文件中
4. **误以为商户子户余额明细可通过来账明细查询获取**：来账明细查询接口仅能查询平台清算专户（9903开头）的入金明细，商户余额明细需通过资金账单下载申请接口获取
5. **误以为退款订单也需要上送网商**：网商不需要退款单上送，系统中直接将状态变更为"已上送"即可，不调接口
